4-1. Capacity Planning — 사이징 공식과 베스트 프랙티스

📌 한 줄 요약

"몇 대가 필요한가?"는 예상 QPS × 응답시간 SLA × 스캔/코어 처리량으로 역산한다. 거기에 메모리·디스크·네트워크 제약을 차례로 검증해 최종 노드 수가 결정된다.

CPU 용량 산출 공식 (공식 블로그 권장)

e_core = (scan_rows / cal_rows) / e_rt * e_qps
변수의미예시
scan_rows한 쿼리가 스캔할 raw 행 수30,000,000
cal_rows코어당 초당 처리 가능한 행 수 (10M~100M)50,000,000
e_rt응답시간 SLA (초)0.3
e_qps예상 동시 QPS180
e_core필요한 vCPU360 vCPU

예시: 30M rows × (1/50M rows/s/core) ÷ 0.3s × 180 QPS = 360 vCPU. 안전 계수(burst 대응) 1.3~1.5배를 곱하면 약 500 vCPU. 노드당 32 vCPU라면 약 16대 BE.

⚠️ 주의 — cal_rows의 폭이 큼

10M~100M rows/s/core는 쿼리 형태에 따라 큰 차이가 난다. 단순 필터/스캔은 100M에 가깝고, 복잡 JOIN/집계/문자열은 10M에 가깝다. 운영 워크로드와 비슷한 SQL로 PoC 측정한 cal_rows를 쓰는 것이 가장 정확.

BE 디스크 한계 (공식 권장)

  • 단일 BE 노드 디스크 합계: 최대 10 TB — 그 이상이면 회복(클론) 시간이 너무 오래 걸려 위험.
  • 디스크 1개당 용량: 2 TB 미만 권장 — IO 병렬도 + 복구 단위 제어.
  • 디스크 종류: SSD/NVMe — HDD는 콜드 매체로만.
  • RAID 미권장 — SR이 자체적으로 디스크들을 균등 분산 사용. RAID 0/5는 단일 실패가 BE 전체로 번짐.
  • 한 디스크당 한 마운트 + ext4/xfs.
구성OK이유

메모리 예산

BE 메모리는 다음 항목에 분배(8장 튜닝·SYS-3 참고).

Query Pool         : 활성 쿼리 (mem_limit의 ~50%)
Storage Page Cache : 컬럼 페이지 캐시 (~20%)
Data Cache         : 외부 카탈로그 캐시 (~10%)
PK Index           : Primary Key 테이블 인덱스
Compaction         : 컴팩션 워커
OS / 여유          : ~20% (커널, free 등)

대략적 메모리 산식:

BE_mem ≈ vCPU × 4 GB                         -- 비율 기준
       + tablet_count × 2 KB                  -- 메타 부담
       + active_query × avg_query_mem         -- 쿼리 풀

네트워크

  • 10 GbE 이상 권장 — 1 GbE는 셔플/적재가 즉시 병목.
  • BE ↔ BE는 같은 스위치/랙이 이상적 — Exchange(셔플) 트래픽이 큼.
  • BE ↔ HMS/HDFS — 외부 카탈로그 사용 시 NameNode/Metastore 도달 가능.
  • BE ↔ Object Storage — Shared-data 모드 또는 Hive on S3.
  • 방화벽: 포트 표 참고. 9030/8030/9010/9020/8040/9050/9060/8060.

Load Balancer (FE 앞단)

운영에서는 FE 3대를 그대로 BI/애플리케이션에 노출하지 않고 LB 뒤에 둔다.

도구장점비고
Nginx (stream)가장 단순, TCP L4health check + retry
HAProxy운영 도구 풍부, 메트릭 좋음가장 흔한 선택
F5 / 자체 LB대기업 표준이미 운영 중이라면 그대로

frontend sr_mysql
  bind *:9030
  mode tcp
  default_backend sr_fe

backend sr_fe
  mode tcp
  balance leastconn
  option tcp-check
  tcp-check connect
  server fe1 fe1.internal:9030 check inter 3s
  server fe2 fe2.internal:9030 check inter 3s
  server fe3 fe3.internal:9030 check inter 3s

동질적 클러스터 (Homogeneous)

모든 BE를 같은 CPU/메모리/디스크로 구성하는 것이 표준. 이질적 BE가 섞이면:

  • 스케줄러가 가장 느린 BE에 맞춰져 전체가 느려짐.
  • Tablet 분배가 불균형해져 디스크 사용 편차 발생.
  • 복제 배치 정책이 복잡해짐.

프로덕션 최소 규모

  • 최소: FE 3 + BE 3 (총 6 노드).
  • 이보다 작으면 단일 BE 다운에서 복제본 손실 위험 또는 FE quorum 깨짐.
  • 학습/개발은 Allin1 컨테이너(P-1)으로 충분.

스케일링 전략

  1. 수직 확장(Vertical) — 같은 BE에 디스크/메모리 추가. 디스크 추가가 가장 흔한 첫 번째 카드.
  2. 수평 확장(Horizontal) — BE 추가 (5장 운영 참고). Tablet 자동 리밸런스.
  3. Compute 확장 only — Shared-data 모드의 CN 추가 (5-3).
  4. Resource Group 분리 — 노드를 늘리지 않고도 워크로드 격리로 체감 성능 향상.

실제 예시 — 약 20대 PM 환경 사이징

본 가이드 환경에 위 공식을 적용해보면:

  • FE 3대 — 별도 PM. 사이징 무관(메타·연결만).
  • BE 17대. 노드당 32~64 vCPU 가정 → 클러스터 총 ~544~1088 vCPU.
  • 코어당 5천만 rows/s 가정. 가능 처리량 = 27 ~ 54 billion rows/s.
  • SLA 0.5초, QPS 200, 평균 scan 5천만 rows라면 → 필요 코어 (5천만/5천만)/0.5×200 = 400 vCPU. 안전 계수 1.3 = 520 vCPU. ✅ 여유.
  • 스토리지: 노드당 SSD 3.84TB × 17 = 65 TB raw / 3복제 = ~22 TB 가용. 이 환경의 단일 BE 디스크 합계 3.84TB < 10 TB ✅.
  • 메모리: 노드당 256GB → mem_limit 80% = 200GB. CPU:메모리 비율이 매우 넉넉(=BE에 유리).
클러스터 적용 노트
  • cal_rows는 운영 워크로드와 비슷한 SQL 10~20개로 PoC 측정해 팀 표준값을 가지고 있는 게 가장 정확. 그 수치 없이는 산식이 수십 배 차이까지 날 수 있음.
  • SSD 8개 / 480GB 구성은 공식 권장(디스크당 < 2TB, 합계 < 10TB)에 잘 부합 — 그대로 진행 권장.
  • HAProxy LB를 처음부터 두면 추후 FE 3 → 5 확장 시 BI/도구 설정 변경이 필요 없음.
  • 이질적 BE를 섞지 말 것 — 약 17대를 동시 발주해 같은 사양으로.
