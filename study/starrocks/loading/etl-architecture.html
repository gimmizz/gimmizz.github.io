ETL 아키텍처 — StarRocks로 모든 ETL을 처리하는 패턴

📌 한 줄 요약

StarRocks 한 엔진으로 Ingest → Stage → Mart → Serve 전체를 처리하는 모던 ETL 아키텍처. 외부 변환 엔진(Spark/Flink) 의존 없이 SQL과 MV로 끝난다.

① Ingest — 소스에서 SR로

소스 종류별로 적재 방식 선택. 적재 개요 표 참고.

  • OLTP(MySQL/PG) → Flink CDC + Routine Load 또는 Stream Load 어댑터
  • 이벤트 스트림(Kafka) → Routine Load (또는 Kafka Connect)
  • 대용량 파일(HDFS/S3) → Pipe (자동 감지) 또는 Broker Load
  • 외부 RDB 정기 동기 → JDBC Catalog + INSERT INTO ... SELECT
  • 외부 데이터 레이크 → Hive/Iceberg/Unified Catalog → INSERT INTO ... SELECT

② Stage — 정제 전 raw 보존

입력을 바로 Mart에 쓰지 않고 1단계 stage 테이블에 그대로 적재한다. 이유:

  • 스키마 변경/소스 오류 시 재처리 가능 (raw가 살아있어서)
  • Mart 변환 로직과 적재를 분리 — 재배치 자유
  • Append-only이므로 가장 단순한 모델(Duplicate)을 사용
CREATE TABLE stg.orders_raw (
  ts          DATETIME NOT NULL,
  order_id    BIGINT,
  user_id     BIGINT,
  status      VARCHAR(32),
  amount      DECIMAL(18,2),
  payload     JSON
)
DUPLICATE KEY(ts, order_id)
PARTITION BY date_trunc('day', ts)
DISTRIBUTED BY RANDOM
PROPERTIES (
  "replication_num" = "3",
  "compression" = "ZSTD",
  "dynamic_partition.enable" = "true",
  "dynamic_partition.start" = "-30",
  "dynamic_partition.end"   = "3",
  "dynamic_partition.time_unit" = "DAY"
);

③ Mart — 변환 결과의 진실 소스

비즈니스 의미를 가진 마트. 변경이 잦으면 Primary Key 모델로 UPSERT/DELETE를 받는다.

CREATE TABLE mart.orders (
  order_id   BIGINT NOT NULL,
  user_id    BIGINT,
  status     VARCHAR(32),
  amount     DECIMAL(18,2),
  updated_at DATETIME
)
PRIMARY KEY(order_id)
PARTITION BY date_trunc('day', updated_at)
DISTRIBUTED BY HASH(order_id)
PROPERTIES (
  "replication_num"         = "3",
  "enable_persistent_index" = "true",
  "dynamic_partition.enable"= "true",
  "dynamic_partition.start" = "-180",
  "dynamic_partition.end"   = "3",
  "dynamic_partition.time_unit" = "DAY"
);

④ Serve — Async MV로 사용자 응답 가속

사용자 SQL을 그대로 두고, 옵티마이저가 MV로 자동 재작성하게 한다.

CREATE MATERIALIZED VIEW mart.mv_orders_d
PARTITION BY dt DISTRIBUTED BY HASH(user_id)
REFRESH ASYNC EVERY (INTERVAL 10 MINUTE)
PROPERTIES ("enable_query_rewrite" = "true")
AS
SELECT date_trunc('day', updated_at) dt, status, count(*) cnt, sum(amount) amt
FROM mart.orders GROUP BY 1, 2;

변환 (Stage → Mart) 표준 패턴

패턴 A — 단순 매핑

INSERT INTO mart.orders
SELECT order_id, user_id, status, amount, updated_at
FROM stg.orders_raw
WHERE ts >= '{{ ds }}'::DATETIME AND ts < '{{ next_ds }}'::DATETIME;

패턴 B — JSON 추출 + 정제

INSERT INTO mart.orders
SELECT
  get_json_int(payload, '$.id')                    AS order_id,
  get_json_int(payload, '$.user.id')               AS user_id,
  COALESCE(get_json_string(payload, '$.status'), 'unknown') AS status,
  CAST(get_json_string(payload, '$.amount') AS DECIMAL(18,2)) AS amount,
  ts AS updated_at
FROM stg.orders_raw
WHERE ts::DATE = '{{ ds }}';

패턴 C — 차원 결합

INSERT INTO mart.events_user
SELECT e.event_time, e.user_id, e.event_type, u.grade, u.country
FROM stg.events_raw e
LEFT JOIN dim.user u ON e.user_id = u.user_id
WHERE e.event_time::DATE = '{{ ds }}';

패턴 D — INSERT OVERWRITE (멱등 백필)

INSERT OVERWRITE mart.orders_d PARTITION (dt='{{ ds }}')
SELECT date_trunc('day', updated_at) dt, user_id, status, count(*) cnt, sum(amount) amt
FROM mart.orders WHERE updated_at::DATE = '{{ ds }}'
GROUP BY 1, 2, 3;

오케스트레이션 — Dagster + dbt 또는 Airflow

Stage → Mart의 INSERT는 dbt 모델로 정의하고, Dagster가 적재 잡(Routine Load의 RUNNING 확인 → dbt run → MV refresh)을 자동화. 자세한 패턴은 외부 도구 연동 섹션 참고.

  • Dagster + dbt
  • dbt-starrocks 어댑터
  • Airflow

외부 카탈로그와 결합

SR-only ETL이라도 raw가 이미 Hive/Iceberg에 있다면 그쪽을 그대로 활용:

-- 외부 카탈로그를 stage로 활용 (SR 자체 적재 X)
INSERT INTO mart.orders
SELECT order_id, user_id, status, amount, updated_at
FROM hive_prod.dwh.orders_raw
WHERE dt = '{{ ds }}';

-- 또는 Async MV가 외부 카탈로그를 베이스로
CREATE MATERIALIZED VIEW mv_hive_orders_d ...
AS SELECT ... FROM hive_prod.dwh.orders_raw GROUP BY ...;

주기작업
실시간(분 단위)Routine Load → mart.orders (PK 자동 UPSERT)
10분 주기mv_orders_minute 자동 refresh
매시 00분mv_orders_hour 자동 refresh

자주 빠뜨리는 운영 포인트

  • label 표준화: {table}_{ds}_{seq} 형식. 재시도 안전.
  • Resource Group 분리: ETL과 BI를 같은 Resource Group에 두면 사용자 쿼리가 ETL 적체 시 영향.
  • 통계 갱신: 큰 INSERT 후 ANALYZE 호출. 옵티마이저가 정확한 플랜을 잡음.
  • MV 의존성: 베이스 테이블 변경 직후엔 MV가 stale. 의도적 큰 변경 후엔 명시적 REFRESH.
  • Stage TTL: stage는 보통 30일이면 충분. 너무 길게 보관하면 디스크 압박.
  • 외부 도구 시점 차이: Routine Load의 적재 완료를 기다리지 않고 dbt run을 실행하면 결과가 비어 있을 수 있음. Dagster sensor로 max(updated_at) 확인 후 실행.
클러스터 적용 노트
  • 약 20대 PM 환경은 ETL+BI 동시 처리에 충분. 단 BI/ETL Resource Group을 처음부터 분리.
  • "Spark/Flink 없이 ETL을 SR로 통합"하는 가장 큰 가치는 운영 변수 감소. 변환·적재·서빙이 한 SQL로 통일.
  • 대용량 외부 파일이 정기적으로 떨어지는 환경이면 Pipe가 가장 자연스러운 적재 방식.
  • 실시간 CDC + 정기 MV는 가장 흔한 조합. 실시간 대시보드 패턴도 같이 참고.
