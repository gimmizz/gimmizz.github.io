Spark Load — TB급 비동기 배치 적재

📌 한 줄 요약

Spark 클러스터를 활용해 정렬·인덱스 빌드까지 사전 처리한 결과물을 SR에 적재. 수십 GB ~ TB 단위 일배치/백필에 적합.

언제 쓰나

  • Hive 또는 HDFS의 매우 큰 테이블 한 번에 적재.
  • SR로의 직접 INSERT가 부담되는 분량.
  • 이미 Spark 클러스터가 있고 변환 로직이 PySpark/Scala로 작성되어 있는 경우.

스트리밍/소량 적재는 적합하지 않다. 그쪽은 Stream Load / Routine Load.

동작 흐름

사전 준비

  1. Spark 클러스터 (YARN 또는 K8s)
  2. Spark 클라이언트가 SR FE/BE에 접근 가능
  3. SR spark 자원 등록: 
    CREATE EXTERNAL RESOURCE 'spark_etl'
PROPERTIES (
  "type"           = "spark",
  "spark.master"   = "yarn",
  "spark.submit.deployMode" = "cluster",
  "working_dir"    = "hdfs://nn/tmp/sr-spark",
  "broker"         = "broker_default"
);

GRANT USAGE_PRIV ON RESOURCE 'spark_etl' TO 'etl_user';

LOAD 잡

LOAD LABEL mart.orders_20260507_full (
  DATA INFILE("hdfs://nn/warehouse/orders/dt=2026-05-07/*")
  INTO TABLE orders
  FORMAT AS "parquet"
  (order_id, user_id, status, amount, updated_at)
)
WITH RESOURCE 'spark_etl' (
  "spark.executor.memory" = "8g",
  "spark.executor.instances" = "20"
)
PROPERTIES (
  "timeout" = "14400",
  "max_filter_ratio" = "0.01"
);

-- 진행 상황
SHOW LOAD WHERE LABEL = 'mart.orders_20260507_full'\G

주의

  • Spark Load는 SR 4.x에서도 유효하나, 최근에는 외부 카탈로그(Hive) + INSERT INTO ... SELECT나 Pipe가 더 자주 쓰임.
  • ETL 중간 산출물이 working_dir(HDFS)에 쌓이므로 정기 청소 필요.
  • Schema 변환·복잡한 윈도우 함수 등은 Spark에서 끝내고 결과만 SR로 보내는 패턴이 일반적.
클러스터 적용 노트
  • "SR-only ETL"을 지향한다면 Spark Load는 거의 쓸 일이 없습니다 — Hive Catalog + INSERT INTO ... SELECT 또는 Pipe가 더 단순합니다.
  • 이미 Spark 클러스터가 있고 변환 로직이 거기 있는 환경에서만 검토.
