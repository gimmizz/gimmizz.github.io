label — 멱등성의 핵심

모든 적재 잡(Stream/Routine/Broker/Spark/INSERT)은 label을 가진다. 같은 label은 한 번만 성공한다.

# 같은 label로 두 번 호출 → 두 번째는 "label already exists"
curl -H "label:orders_20260507_001" ...
curl -H "label:orders_20260507_001" ...   # ← 무시 / 충돌

-- INSERT도 label 지정 가능
INSERT INTO mart.orders WITH LABEL orders_20260507_001 SELECT ...;

label 표준: {table}_{yyyymmddHHMMSS}_{seq} 또는 {table}_{ds}_{run_id}. ETL 도구의 run id를 끼워 넣으면 재시도 안전.

max_filter_ratio — 부분 실패 허용

# 기본 0 — 한 행이라도 실패하면 잡 실패
-H "max_filter_ratio:0"

# 1% 까지 실패 허용 (예: 일부 잘못된 행이 섞이는 케이스)
-H "max_filter_ratio:0.01"

운영에서는 0.001 ~ 0.01 정도가 무난. 허용 비율을 너무 크게 두면 데이터 품질 저하 감지가 늦어진다.

ErrorURL — 실패 행 확인

적재가 실패하거나 일부 필터된 경우 응답에 ErrorURL이 포함됨. 그 URL을 따라가면 실패한 행들이 텍스트로 저장되어 있다.

# 응답 JSON 예
{
  "Status": "Success",
  "Message": "OK",
  "NumberFilteredRows": 3,
  "ErrorURL": "http://be1:8040/api/_load_error_log?file=__shard_0/error_log_xxx"
}

curl http://be1:8040/api/_load_error_log?file=...    # 실패 행 보기

Routine Load는 SHOW ROUTINE LOAD ...의 ErrorLogUrls 필드.

Strict Mode

입력 행의 컬럼 변환 결과가 NULL인데 NOT NULL 컬럼으로 들어가는 경우 등을 어떻게 처리할지 결정.

-- Stream Load 헤더
-H "strict_mode:true"

# strict_mode = true:  변환 실패 행을 즉시 fail (또는 max_filter_ratio에 카운트)
# strict_mode = false: NULL을 그대로 INSERT 시도 (NOT NULL 컬럼이면 또 다른 실패)

운영에서는 strict_mode = true + max_filter_ratio = 0.001 조합이 자주 쓰임.

timeout

-H "timeout:600"           # 600초 (Stream Load 기본 600s)
-- Broker/Spark Load: PROPERTIES("timeout"="14400")  (4시간 등)

대용량 잡은 timeout을 충분히. 타임아웃이 되면 잡은 CANCELLED.

재시도 패턴

  1. 같은 잡 같은 label로 재호출 → label 충돌이라 두 번 처리 안 됨 (멱등).
  2. label만 새로 만들어 재시도 → 두 번 처리되니 적재 측에서 중복 발생 가능. PK 테이블이면 UPSERT라 안전.
  3. 일반 패턴: ETL 도구가 task instance ID를 label에 끼워 넣어 자동 멱등.

Routine Load 실패 정책

PROPERTIES (
  "max_error_number"     = "10000",   -- 누적 허용 실패 행
  "max_filter_ratio"     = "0.01",    -- 한 batch 허용 실패 비율
  "max_batch_interval"   = "10",
  "max_batch_rows"       = "200000",
  "max_batch_size"       = "104857600"
)

실패 행이 한도를 넘으면 잡이 PAUSED 상태로 멈춘다. Routine Load Runbook에서 RESUME 절차 참조.

진단 흐름

  1. SHOW LOAD WHERE LABEL='...'의 ErrorMsg + ErrorURL.
  2. ErrorURL에서 실패 행 직접 확인.
  3. 표현식·jsonpaths·column 매핑 점검.
  4. 스키마 (NOT NULL, DEFAULT) 점검.
  5. fe.log / be.WARNING에서 같은 시각 메시지.

Stream Load Transaction (2PC)

여러 적재 호출을 묶어 atomic하게 commit/rollback하고 싶을 때.

# 1) Begin
curl -H "label:tx_001" -X POST .../api/transaction/begin

# 2) 데이터 N번 PUT (같은 label)
curl -H "label:tx_001" -T part1.csv ...
curl -H "label:tx_001" -T part2.csv ...

# 3) Prepare → Commit (또는 Rollback)
curl -H "label:tx_001" -X POST .../api/transaction/prepare
curl -H "label:tx_001" -X POST .../api/transaction/commit

Flink Connector의 exactly-once 모드가 내부적으로 이 방식을 사용.

클러스터 적용 노트
  • label 충돌·실패는 ETL 알람의 1순위 — 자동화 안 해두면 사고가 늦게 발견.
  • max_filter_ratio와 strict_mode는 신중히. 너무 관대하면 데이터 품질 저하가 묻힌다.
  • ErrorURL을 ETL 로그에 자동 기록 — 사후 분석에 필수.
