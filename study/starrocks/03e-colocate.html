3-4. Colocate Group — 셔플 없는 JOIN

📌 한 줄 요약

같은 키로 자주 JOIN하는 테이블들을 같은 Colocate Group에 묶으면, BE는 같은 Tablet ID끼리만 로컬 조인해 셔플이 0이 된다. 가장 빠른 조인 전략.

동작 원리

graph LR A1["a Tablet 1
@ BE1"] A2["a Tablet 2
@ BE2"] B1["b Tablet 1
@ BE1"] B2["b Tablet 2
@ BE2"] A1 --- B1 A2 --- B2 R["Local Join, No Shuffle"] A1 --> R A2 --> R B1 --> R B2 --> R

그림 3-4-1. 같은 colocate group의 같은 bucket id는 같은 BE에 위치

조건 (이게 모두 일치해야 한 그룹)

  • 분배 키의 컬럼 수와 데이터 타입이 동일
  • 버킷 수가 동일
  • 복제본 수가 동일

사용

-- 두 테이블을 같은 그룹 'g_user'에 묶음
CREATE TABLE orders (
  order_id BIGINT, user_id BIGINT, ...
)
PRIMARY KEY(order_id)
DISTRIBUTED BY HASH(user_id) BUCKETS 32
PROPERTIES ("colocate_with" = "g_user");

CREATE TABLE user_dim (
  user_id BIGINT, name VARCHAR(64), ...
)
PRIMARY KEY(user_id)
DISTRIBUTED BY HASH(user_id) BUCKETS 32
PROPERTIES ("colocate_with" = "g_user");

상태 확인

SHOW PROC '/colocation_group';

-- 특정 그룹 상세
SHOW PROC '/colocation_group/<group_id>';

주의사항

  • colocate group의 균형이 깨지면 자동으로 재배치(rebalance)가 일어남. 잠시 쿼리 영향 가능.
  • BE 추가/제거 시 colocate group 수가 많으면 리밸런스 부담이 커짐.
  • 분배 키가 다르면 같은 그룹에 못 들어감 — JOIN 키를 미리 일관되게 설계해야 함.
클러스터 적용 노트
  • 차원 모델에 강력: fact 테이블과 dim 테이블의 키를 일치시키고 같은 Colocate Group에 두면 BI 응답이 즉시 빨라짐.
  • 그룹은 적게: 약 17 BE 환경에선 너무 많은 colocate group은 리밸런스 비용을 부담. 핵심 결합 3~5개 그룹 정도가 무난.
  • 새 BE 추가 시: colocate group이 많을수록 데이터 이동량이 많음 → 야간에 BE 1대씩 천천히 추가.
