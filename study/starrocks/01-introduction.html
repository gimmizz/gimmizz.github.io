1. StarRocks 소개

📌 한 줄 요약

StarRocks는 실시간·다차원·고동시성 분석을 위한 오픈소스 MPP 분석 데이터 웨어하우스다. MySQL 프로토콜 호환 + 데이터 레이크 직접 쿼리(External Catalog) + Async Materialized View가 핵심 무기.

1.1 StarRocks가 뭔가

StarRocks는 "차세대 고성능 분석용 데이터 웨어하우스"로 자신을 소개합니다. OLAP(Online Analytical Processing) 워크로드를 빠르게 처리하기 위해 설계된 MPP(Massively Parallel Processing) 엔진이고, 라이선스는 Apache 2.0입니다. 문서 작성 시점 최신 안정 버전은 4.1이며, 3.1 ~ 4.x 메이저 라인이 동시 유지되고 있습니다.

DuckDB·ClickHouse·Apache Doris·Trino처럼 "분석 쿼리"에 특화된 엔진들과 같은 카테고리에 속하지만, StarRocks는 다음과 같은 네 가지 강점을 동시에 내세웁니다.

  1. 네이티브 MPP + 완전 벡터화 실행 엔진 — CPU 캐시·SIMD를 적극 활용해 단일 노드 성능을 끌어올림.
  2. 실시간 업데이트 가능한 칼럼 스토리지 — 컬럼형 OLAP인데도 Primary Key 모델로 UPSERT/DELETE 지원.
  3. 비용 기반 옵티마이저(CBO) — 통계 기반의 조인 순서/플랜 선택.
  4. External Catalog — 데이터를 옮기지 않고도 Hive·Iceberg·Hudi·Delta·MySQL·PostgreSQL 등을 직접 쿼리.

1.2 왜 만들어졌나 — OLAP의 변화

전통적으로 OLAP 시장은 다음과 같이 분화되어 있었습니다.

  • MOLAP(Pre-aggregation 큐브) — Kylin 류. 미리 큐브를 만들어 빠르지만 유연성이 낮음.
  • ROLAP(쿼리 시 집계) — Presto/Trino 류. 유연하지만 큐브식 응답 속도는 못 따라옴.
  • 실시간 OLAP — Druid/Pinot. 실시간 ingestion·집계엔 강하나 복잡한 조인이 약함.
  • HTAP / 칼럼 DBMS — ClickHouse/Doris. 단일 테이블엔 강하나 다수 카탈로그 연계가 약하던 시절이 있었음.

StarRocks는 이 장점들을 한 엔진에 모으려는 시도입니다. "하나의 엔진으로 대시보드·임시쿼리·데이터 레이크 분석·실시간 분석을 모두 다룬다"가 마케팅 문구이자 설계 방향입니다.

1.3 핵심 특징

네이티브 MPP + 벡터화

StarRocks의 BE(Backend) 노드들은 쿼리를 잘게 쪼개 동시에 처리합니다(MPP). 엔진은 C++로 작성되었고, 모든 연산자가 벡터화(컬럼 단위 처리) 되어 있어 한 행씩 도는 전통적 처리에 비해 캐시 미스/분기 예측 비용이 훨씬 작습니다.

CBO (Cost-Based Optimizer)

StarRocks 옵티마이저는 통계(NDV, Row Count, 데이터 분포 등)를 기반으로 조인 순서·조인 알고리즘(Broadcast/Shuffle/Bucket-Shuffle/Colocate)·집계 위치를 선택합니다. 특히 별 모양(스타) 스키마의 다중 조인에서 안정적인 성능을 냅니다.

Materialized View (특히 Async MV)

StarRocks의 MV는 두 종류입니다.

  • Sync MV — 베이스 테이블과 동시에 갱신되는 단일 테이블 집계 인덱스. 가볍고 빠름.
  • Async MV — 다수 테이블·External Catalog까지 결합 가능하고 스케줄/파티션 기반 리프레시. 쿼리 자동 재작성(query rewrite)이 켜져 있어, 사용자는 원본 쿼리를 그대로 쓰지만 옵티마이저가 알아서 MV를 활용합니다.

실시간 업데이트 (Primary Key Table)

Primary Key 테이블은 컬럼 스토리지인데도 UPSERT/DELETE를 지원하기 위해 Delete-and-Insert 전략 + 메모리에 PK 인덱스를 둡니다. Flink CDC, Kafka, Routine Load 등으로 들어오는 변경 이벤트를 거의 실시간으로 반영하면서 OLAP 쿼리도 함께 받을 수 있습니다.

External Catalog (데이터 레이크 직접 분석)

Hive Metastore/Glue/AWS·HDFS·S3·Iceberg·Hudi·Delta·JDBC·Elasticsearch·Paimon 등을 외부 카탈로그로 마운트하면, StarRocks 안에서 마치 자체 테이블처럼 SQL을 쓸 수 있습니다. "데이터를 옮기지 않는 OLAP" 즉 레이크하우스 쿼리 엔진으로서의 사용이 가능해집니다.

MySQL 프로토콜 호환

StarRocks의 FE(Frontend)는 MySQL 와이어 프로토콜을 그대로 말합니다. 그래서 MySQL CLI(mysql -h ... -P 9030 -u root)·JDBC·Python mysql-connector·DBeaver/Tableau/Superset 등 MySQL을 다루던 도구를 그대로 사용할 수 있습니다.

1.4 주요 사용 사례

  • 실시간 대시보드 — Flink/Kafka로 들어오는 데이터를 Primary Key 테이블에 적재 후 BI에서 즉시 조회.
  • 고동시성 광고/추천 분석 — 같은 쿼리를 동시에 수백·수천이 두드리는 워크로드.
  • 임시(ad-hoc) 분석 — 분석가가 콘솔에서 자유롭게 조인·필터링.
  • 레이크하우스 쿼리 엔진 — Hive/Iceberg/Hudi/Delta 위에서 Trino 대안으로 사용.
  • 통합 SQL 게이트웨이 — JDBC Catalog로 MySQL/PostgreSQL/Oracle을 묶어 단일 SQL 인터페이스 제공.

1.5 다른 엔진과의 위치

Apache Doris StarRocks와 같은 뿌리(Apache Doris fork에서 분기) 커뮤니티 노선 차이 실질적 사촌. CBO/MV/Lakehouse 기능에서 StarRocks가 좀 더 공격적으로 발전

1.6 버전 정책

StarRocks는 메이저 버전(3.1, 3.2, 3.3, 3.4, 3.5, 4.0, 4.1 ...)을 병렬로 유지보수합니다. 한 번 채택한 메이저 라인을 따라가는 게 운영 안정성 측면에서 유리합니다. 마이너 패치는 비교적 자주 나오므로 보안/버그픽스 추적이 필요합니다.

  • 4.x — 최신 라인. Async MV 자동 재작성 강화, Lakehouse 기능, Shared-data 모드 안정화.
  • 3.x — LTS 격으로 폭넓게 사용. 3.2부터 외부 카탈로그·MV 기능이 크게 개선됨.
클러스터 적용 노트 — 약 20대 PM 환경에서의 의미

우리 환경은 온프레미스 자체 데이터센터이고, 외부 카탈로그(Hive/HDFS/MySQL 등)를 다수 붙일 예정이라 "Trino 또는 Doris 대신 StarRocks를 쓰는가?"라는 질문이 자연스럽습니다. 실용적인 결론은:

  • 외부 카탈로그를 한 엔진에서 묶으면서, 자체 테이블에 자주 쓰는 결과는 Async MV로 캐싱하고 싶다 → StarRocks가 적합.
  • SSD 8개 × 약 20대라는 구성에서 가장 큰 가치는 로컬 스토리지가 충분하다는 점이고, 따라서 우선은 Shared-nothing 모드로 시작하는 게 자연스럽습니다 (Shared-data 모드는 객체 스토리지 도입과 함께 나중에 검토).
  • 관리해야 할 외부 시스템(ZK, BookKeeper 등)이 없다는 점이 운영 인력 절약에 직접 도움이 됩니다 — FE 자체가 BDB JE 기반의 메타데이터 저장소를 내장합니다.
