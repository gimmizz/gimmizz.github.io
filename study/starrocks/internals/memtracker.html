SYS-3. MemTracker — BE 메모리 추적

BE는 메모리 사용을 계층적 MemTracker로 추적한다. OOM 방지·디버깅의 출발점.

계층

graph TD PROC["process
= mem_limit"] PROC --> Q["query 풀"] PROC --> L["load 풀"] PROC --> C["compaction"] PROC --> S["storage page cache"] PROC --> DC["data cache"] PROC --> PK["pk index"] PROC --> META["metadata 등"] Q --> Q1["query #1"] Q --> Q2["query #2 (각 fragment 별 child tracker)"]

상태 확인

-- BE 메모리 풀별 사용량
SELECT BackendId, IP, MemUsedPct, AvailMemBytes
FROM information_schema.backends;

-- 더 자세히 — BE Web UI
http://<be>:8040/mem_tracker

-- HTTP API
curl http://be1:8040/api/memory/info | jq

메모리 한도 도달 시 동작

  1. 각 쿼리 child tracker는 query_mem_limit 까지만 허용.
  2. 한 쿼리가 한도를 넘으면 그 쿼리만 Mem limit exceeded로 실패.
  3. process 전체가 mem_limit에 가까워지면 spill 또는 새 쿼리 거부.
  4. OOM Killer까지 가면 BE 자체가 종료 — 절대 피해야 함.

Disk Spill (v3.x)

큰 쿼리가 메모리를 초과할 때 일부 연산자(Sort, Hash Aggregate, Hash Join)는 디스크로 spill 가능.

SET enable_spill = true;
SET spill_storage_limit = '50G';

OLTP-like 응답성 SLA가 중요한 환경에선 spill을 끄고 차라리 fail-fast하는 게 나을 수 있음.

튜닝 권장

파라미터의미권장
mem_limitBE 전체 한도OS의 80%
query_mem_limit (세션)한 쿼리 메모리 한도32~64GB
load_mem_limit적재 잡 메모리 한도적재 패턴 따라
storage_page_cache_limit페이지 캐시20~30%
compaction_mem_limit컴팩션10%
datacache_mem_size외부 카탈로그 캐시 메모리부5~10%

OOM 발생 시

  1. be.WARNING / OS dmesg에서 OOM 이벤트 확인.
  2. 메모리 풀별 사용량을 BE Web UI에서 추출 — 어디가 폭증했는지.
  3. 큰 쿼리가 원인이면 query_mem_limit로 fail-fast하도록.
  4. 적재 폭주가 원인이면 batch 크기/주기 조정.
  5. Page Cache가 너무 크면 storage_page_cache_limit 낮춤.
클러스터 적용 노트
  • 256GB BE의 표준 분배: query 100GB, page cache 50GB, data cache 30GB, compaction 20GB, 기타 20GB, OS 36GB.
  • 큰 ad-hoc 쿼리를 막으려면 사용자별 query_mem_limit을 Resource Group에서 차등 부여.
  • Spill은 분석 워크벤치에서만 ON, BI/대시보드는 OFF로 fail-fast — SLA 안정성에 도움.
