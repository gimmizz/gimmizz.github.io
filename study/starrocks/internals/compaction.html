SYS-1. Compaction — 컴팩션의 종류와 동작

왜 필요한가

BE는 적재가 들어올 때마다 새 RowSet(=Segment 그룹)를 만든다. 시간이 지나면 한 Tablet에 RowSet이 누적되어 쿼리 시 머지 비용이 커진다. Compaction은 작은 RowSet들을 모아 큰 RowSet으로 통합하는 백그라운드 작업.

graph LR L1["새 적재 = 작은 RowSet 다수"] L1 --> CC["Cumulative Compaction
최근 RowSet들을 한 RowSet으로"] CC --> BC["Base Compaction
오래된 base + cumulative을 통합"] PK["PK Update Compaction
(PK 테이블 전용)"]

3 가지 종류

종류대상동작
Cumulative최근 작은 RowSet 다수한 cumulative RowSet으로 합침. 가장 자주 일어남.
Basecumulative + 기존 base가장 큰 base RowSet으로 통합. 비용 큰 작업.
Update (PK 테이블)delete-vector + basedelete 마커를 base에 반영. PK 테이블 쓰기 후 정리.

Compaction Score

각 Tablet에 부여되는 적체 지표. RowSet 수와 크기 분포를 기반. 일반적으로 100을 넘으면 적체 알람.

SELECT BackendId, IP, MaxCompactionScore
FROM information_schema.backends
ORDER BY MaxCompactionScore DESC LIMIT 10;

관련 BE 파라미터

# be.conf
# 디스크당 워커 수
base_compaction_num_threads_per_disk        = 1
cumulative_compaction_num_threads_per_disk  = 1
update_compaction_num_threads_per_disk      = 1     # PK 테이블

# 동시 cumulative compaction 최대
max_cumulative_compaction_num_singleton_deltas = 1000

# 메모리 한도
compaction_mem_limit = 10%

# 우선순위
cumulative_compaction_skip_window_seconds = 30      # 최근 30초 이내 RowSet은 건너뜀

수동 트리거 (긴급 시)

-- 특정 Tablet 강제 compaction
ADMIN SET BACKEND TABLET COMPACTION TYPE = "BASE" FOR TABLET 12345;
-- 또는 HTTP API
curl -X POST http://be1:8040/api/compaction/run?tablet_id=12345&compaction_type=base

자주 적체되는 패턴

  • 너무 작은 batch로 빈번하게 적재 (Stream Load 1초마다 등) → Cumulative 적체.
  • PK 테이블에 UPSERT가 폭주 → Update Compaction 부족.
  • 디스크당 워커가 1인데 워크로드 증가 → 워커 수 늘리기.
  • BE 메모리 부족 → compaction이 throttling.
클러스터 적용 노트
  • 약 17 BE × 디스크 8 환경에서 PK 테이블이 많다면 update_compaction_num_threads_per_disk = 2부터 시작해 모니터링.
  • 적재 batch는 너무 작게 자주 보내지 말 것 — 한 번에 5~10초/수만 행 단위가 안정적.
  • Compaction이 쿼리 IO와 경쟁 — 워커 수와 compaction_mem_limit의 trade-off를 모니터링하면서 조정.
