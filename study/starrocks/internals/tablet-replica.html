SYS-4. Tablet 복제·복구

배치 원칙

  • 한 Tablet의 N개 복제본은 서로 다른 BE에 위치.
  • Rack/Zone 인지가 가능하면 다른 rack에 분산 — 더 강한 내고장성.
  • BE의 디스크들 사이에서 라운드로빈으로 균형.

Replica 상태

상태의미
HEALTHY정상
VERSION_MISSING일부 버전 누락 — clone으로 복구
REPLICA_MISSING복제본 수 부족 — 신규 BE에서 clone
UNHEALTHYBE 다운/네트워크 단절
FORCE_REDUNDANT복제본 과잉 — 정리 대상

자동 복구 (Clone)

graph LR BE1["BE1 down"] T["Replica missing 감지"] S["FE Scheduler"] C["Clone Task"] N["새 BE에서 받은 복제본"] BE1 --> T --> S --> C --> N
-- 진행 중인 clone/복구 큐
SELECT type, count(*)
FROM information_schema.tablet_schedules
WHERE state IN ('PENDING','RUNNING')
GROUP BY type;

-- 자세히
SHOW PROC '/cluster_balance';

관련 FE 파라미터

# fe.conf
tablet_repair_delay_factor_second = 60      # 즉시 복구 시작까지 대기
tablet_sched_max_scheduling_tablets = 10000
clone_thread_num = 16
schedule_slot_num_per_path = 2

수동 조정

-- 특정 Tablet 보고
SHOW TABLET 12345\G

-- 특정 BE의 모든 Tablet 이전 (DECOMMISSION)
ALTER SYSTEM DECOMMISSION BACKEND "be05:9050";

-- 잘못된 복제본 일치 강제
ADMIN SET REPLICA STATUS PROPERTIES (
  "tablet_id" = "12345",
  "backend_id" = "10003",
  "status" = "bad"
);
-- bad로 두면 다음 스케줄러 라운드에서 새 복제 생성

리밸런스

BE 추가 시 자동으로 일부 Tablet이 새 BE로 이전된다. 부하가 큰 시점에는 잠시 쿼리 영향이 있을 수 있어 야간 작업 권장.

-- 리밸런스 정지 / 재개
ADMIN SET FRONTEND CONFIG ("disable_balance" = "true");
ADMIN SET FRONTEND CONFIG ("disable_balance" = "false");

Colocate Group 리밸런스

Colocate group은 정합이 깨지면 그룹 단위로 리밸런스 수행. 그룹이 많을수록 BE 추가/제거 비용이 커짐.

SHOW PROC '/colocation_group';
-- 그룹별 stable / unstable 상태 확인
클러스터 적용 노트
  • BE 1대 재기동은 약 17 BE 환경에서 흔한 작업. 재기동 직후 잠시 unhealthy/clone 큐가 쌓이는 건 정상 — 알람은 5~10분 지속 조건.
  • Decommission은 반드시 끝까지 기다린 후 DROP. 중간에 끊으면 복제본 손실 위험.
  • 대량 BE 추가 시 disable_balance를 잠시 켜고 ADD BACKEND를 모두 끝낸 후 한꺼번에 끄는 패턴이 안정적.
