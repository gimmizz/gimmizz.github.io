5-3. Shared-Data 모드 — 스토리지·컴퓨트 분리

📌 한 줄 요약

데이터는 객체 스토리지(S3/MinIO/HDFS)에 두고, CN(Compute Node)은 stateless하게 캐시만 보관. 탄력적 확장과 비용 절감이 가능. 약 20대 PM 환경은 우선 Shared-nothing이 자연스럽지만, 향후 검토 가치가 있음.

그림 5-3-1. Shared-Data 클러스터 구성

Storage Volume 등록

-- S3 호환
CREATE STORAGE VOLUME def
TYPE = S3
LOCATIONS = ('s3://sr-warehouse/')
PROPERTIES (
  "enabled" = "true",
  "aws.s3.endpoint" = "https://minio.internal:9000",
  "aws.s3.access_key" = "...",
  "aws.s3.secret_key" = "...",
  "aws.s3.enable_path_style_access" = "true"
);

SET def AS DEFAULT STORAGE VOLUME;

-- 테이블이 자동으로 객체 스토리지에 저장됨
CREATE TABLE mart.orders (...) ;

캐시 설정

# cn.conf
storage_root_path  = /data/sr/cache    -- 로컬 캐시
datacache_disk_size = "400G"
storage_page_cache_limit = 30%

핫 데이터는 로컬 SSD 캐시에서 직접 스캔, 콜드 데이터는 원격에서 가져오며 동시에 캐시에 채움.

테이블에서 명시적으로 사용

CREATE TABLE mart.events (
  ...
)
DUPLICATE KEY(...)
PARTITION BY ...
DISTRIBUTED BY ...
PROPERTIES (
  "datacache.enable" = "true",
  "datacache.partition_duration" = "30 day",  -- 최근 30일 파티션을 캐시
  "storage_volume" = "def"
);

Shared-Nothing vs Shared-Data 선택 기준

상황권장
저지연·고동시성 BI, 안정 데이터셋Shared-Nothing
탄력적 컴퓨팅(낮/밤 다른 부하)Shared-Data
다수 클러스터가 같은 데이터 공유Shared-Data
SSD 비용 부담 / 보관기간 길다Shared-Data
온프레 + 데이터가 BE SSD 합계 안에 들어감Shared-Nothing
객체 스토리지 운영 부담 대비 탄력성 이점이 작음Shared-Nothing
클러스터 적용 노트 — 본 가이드 환경의 도입 시점
  • 지금은 SSD 약 65TB(원본)이 충분히 크므로 Shared-Nothing이 단순·빠름.
  • 다음 조건이 누적되면 Shared-Data 검토:
    • 환경에 안정적인 S3 호환 객체 스토리지(MinIO 등)가 정착
    • "보관 기간 1년+ 데이터" 비중이 50% 넘어감
    • 야간/낮 컴퓨팅 부하 차이가 큼 → CN 자동 확장 가치 있음
    • 여러 분석 클러스터가 같은 데이터를 보고 싶음
  • 이행 전략: 신규 마트만 별도 Shared-Data 클러스터로 시작 → 안정화 후 점진 이행. 기존 Shared-Nothing은 핫 분석 가속용으로 유지.
