스키마 진화 (ALTER 표준)

변경 종류와 비용

변경비용비고
컬럼 추가 (NULL 허용)매우 작음메타만 변경. 즉시 완료.
컬럼 추가 (DEFAULT 값)작음적재 시점부터 채워짐. 과거 row는 NULL 또는 DEFAULT.
컬럼 삭제작음메타만 변경. 데이터는 백그라운드 정리.
컬럼 이름 변경작음메타만 변경.
컬럼 타입 호환 위젠 (INT→BIGINT)중간Schema Change 잡 발생.
컬럼 타입 변경 (DOUBLE→DECIMAL)큼전체 재작성. 야간 권장.
인덱스 추가/삭제중간BUILD INDEX 또는 컴팩션 통해 적용.
분배 키 변경불가새 테이블 생성 + INSERT INTO ... SELECT.
모델 변경 (Duplicate↔PK 등)불가새 테이블.
파티션 키 변경불가새 테이블.

컬럼 추가

ALTER TABLE mart.orders ADD COLUMN memo VARCHAR(256) DEFAULT '';

-- 위치 지정
ALTER TABLE mart.orders ADD COLUMN memo VARCHAR(256) AFTER status;

-- 여러 컬럼 한 번에
ALTER TABLE mart.orders
  ADD COLUMN memo VARCHAR(256),
  ADD COLUMN tags ARRAY<STRING>;

삭제 / 이름 변경

ALTER TABLE mart.orders DROP COLUMN memo;
ALTER TABLE mart.orders RENAME COLUMN memo to note;

타입 변경

ALTER TABLE mart.orders MODIFY COLUMN amount DECIMAL(20, 4);

-- 진행 상황
SHOW ALTER TABLE COLUMN FROM mart;
SHOW ALTER TABLE COLUMN WHERE STATE != 'FINISHED';

-- 취소 (FINISHED 전)
CANCEL ALTER TABLE COLUMN FROM mart.orders;

인덱스

ALTER TABLE mart.orders ADD INDEX idx_status (status) USING BITMAP;
ALTER TABLE mart.orders SET ("bloom_filter_columns" = "user_id, order_id");
ALTER TABLE mart.orders DROP INDEX idx_status;

-- 기존 데이터에 인덱스 적용 강제
ALTER TABLE mart.orders BUILD INDEX idx_status FOR ALL TABLETS;

파티션 관리

-- 동적 파티션 옵션 변경
ALTER TABLE mart.orders SET (
  "dynamic_partition.start" = "-180",
  "dynamic_partition.end"   = "7"
);

-- 파티션 추가 / 삭제
ALTER TABLE mart.orders ADD PARTITION p20260601 VALUES [('2026-06-01'),('2026-06-02'));
ALTER TABLE mart.orders DROP PARTITION p20260101;

-- 파티션 내용 비우기 (구조 유지)
TRUNCATE TABLE mart.orders PARTITION (p20260101);

불가능한 변경 — 회피 패턴

분배 키·파티션 키·모델은 ALTER로 변경 못함. 새 테이블 + 적재 + 스왑이 표준.

-- 1) 새 테이블 (변경된 분배/파티션/모델)
CREATE TABLE mart.orders_v2 (
  ...
)
PRIMARY KEY(order_id)
PARTITION BY date_trunc('day', updated_at)
DISTRIBUTED BY HASH(user_id)            -- 변경됨
PROPERTIES (...);

-- 2) 백필 (파티션 단위로 끊어서)
INSERT INTO mart.orders_v2 SELECT * FROM mart.orders WHERE updated_at >= '2026-04-01';

-- 3) ETL을 v2로 전환 (Routine Load 등)

-- 4) 검증 후 스왑
ALTER TABLE mart.orders RENAME orders_legacy;
ALTER TABLE mart.orders_v2 RENAME orders;

-- 5) 일정 기간 후 legacy 삭제
DROP TABLE mart.orders_legacy;

MV 영향

  • 베이스 테이블의 컬럼 추가는 일반적으로 MV에 영향 없음.
  • 컬럼 삭제·타입 변경은 MV 정의가 의존하면 INACTIVE로 떨어질 수 있음. SHOW MATERIALIZED VIEW로 확인 후 ACTIVE로 복구 또는 재생성.
  • 가능하면 스키마 변경 전후로 MV 한 번 리프레시해 정합 확인.
클러스터 적용 노트
  • 운영 DDL은 항상 PR/리뷰 후 적용. 잘못된 ALTER 후 리커버리는 비싸다.
  • 큰 타입 변경은 야간 + Resource Group 격리.
  • 분배/파티션 변경이 필요하면 _v2 테이블 패턴이 표준 — RENAME 스왑이 안전한 컷오버.
