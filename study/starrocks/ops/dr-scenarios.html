DR (Disaster Recovery) 시나리오

💡 RTO / RPO 정의

RTO: 장애 발생부터 서비스 복구까지 허용 시간. RPO: 손실 허용 데이터 시점 (예: "마지막 1시간 데이터까지는 손실 OK"). DR은 둘을 명확히 정해 놓고 시나리오별 절차를 검증하는 작업.

시나리오 1 — BE 1대 다운

항목대응
영향해당 BE의 Tablet은 다른 2복제본에서 처리 — 쿼리 정상
자동 복구tablet_repair_delay_factor_second(기본 60초) 후 자동 클론 시작
운영자 대응BE 재기동 시도. 5분 이상 죽어 있으면 클론 큐 모니터링
RTO0 (쿼리 무중단)
RPO0

시나리오 2 — BE 다수 동시 다운

  • 한 Tablet의 모든 복제본이 동시에 죽으면 그 Tablet은 일시 접근 불가.
  • BE 중 하나만 살아나면 나머지가 자동 복구됨.
  • 대응: BE 다수가 같은 rack/스위치에 몰리지 않게 배포 단계에서 분산. 복제본 배치 정책 점검.

시나리오 3 — FE 1~2대 다운

  • FE 1대 다운: Leader 자동 재선출. 영향 거의 없음.
  • FE 2대 다운(=Follower 과반 깨짐): 메타데이터 쓰기 불가. 읽기는 가능.
  • 대응: 죽은 FE 재기동. 빠르게 복구 안 되면 Observer 1대를 임시 Follower로 승격(ALTER SYSTEM ADD FOLLOWER)도 가능.

시나리오 4 — FE 메타데이터 손상 (가장 무서운 케이스)

graph LR D["FE 메타 손상 감지
(BDB JE 부트 실패)"] S["모든 FE 정지"] R["fe-meta 백업 복원
(어제 야간 tar.gz)"] L["FE 1대만 먼저 기동
(Leader 격)"] H["다른 FE는 --helper 로 합류"] V["SHOW FRONTENDS / BACKENDS
alive 확인"] D --> S --> R --> L --> H --> V 
# 절차
1. 모든 FE에서 stop_fe.sh
2. 가장 최신 fe-meta 백업을 같은 위치에 압축 해제
   (백업 정책: 매일 야간 tar 후 별도 NAS/HDFS, 30~90일 보관)
3. 한 FE만 먼저 시작 → 메타가 그대로 살아남
4. 다른 FE에서:
   bin/start_fe.sh --daemon --helper <leader>:9010
5. SHOW FRONTENDS / SHOW BACKENDS alive 확인

RPO ≈ 마지막 fe-meta 백업 시점(보통 0~24h). RTO ≈ 30분~2시간.

시나리오 5 — 테이블 실수 DROP

-- 가장 마지막 SNAPSHOT으로 복원
RESTORE SNAPSHOT mart.snap_20260506
FROM backup_hdfs
ON (orders)
PROPERTIES (
  "backup_timestamp" = "...",
  "replication_num"  = "3"
);

SHOW RESTORE FROM mart\G

예방: DROP TABLE 대신 RENAME orders → orders_legacy_YYYYMMDD 후 며칠 두고 삭제.

시나리오 6 — 사이트 완전 손실

  1. 새 데이터센터/리전에 동일 버전 SR 클러스터 구축 (Ansible 활용).
  2. fe-meta 백업 복원.
  3. BACKUP SNAPSHOT으로 핵심 마트 RESTORE.
  4. 외부 카탈로그(Hive/Iceberg/MySQL)는 원천이 살아 있다면 카탈로그 정의만 다시 만들면 즉시 사용 가능.
  5. Routine Load 잡 재생성 (Kafka offset은 마지막 적재 시점 + α부터).
  6. BI/도구의 connection 호스트 변경 (FE LB 주소).

DR 리허설

  • 분기 1회: 임시 클러스터에 fe-meta + 핵심 SNAPSHOT 복원 → 결과 SQL 비교 → RTO 측정.
  • 월 1회: BE 1대 강제 종료 → 자동 복구 확인.
  • 매주: 백업 잡 성공 여부 자동 검증.
클러스터 적용 노트
  • 가장 큰 약점은 fe-meta. 매일 야간 백업 + 30~90일 보존은 타협 불가.
  • BACKUP은 외부 저장소(별도 장애 도메인)에. 같은 사이트에만 두면 무의미.
  • 외부 카탈로그가 많을수록 DR이 실은 가벼워진다 — 데이터의 책임이 원천에 있어 SR은 카탈로그 정의만 복원하면 됨.
