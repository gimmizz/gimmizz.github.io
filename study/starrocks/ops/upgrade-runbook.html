OPS-3. 업그레이드 절차서 (Runbook)

사전 준비

  1. 릴리스 노트 정독 — 호환성 변경, 메타 마이그레이션 여부.
  2. 현재 버전·새 버전 확인: SELECT version();
  3. FE 메타 디렉터리 백업: tar czf fe-meta-$(date +%F).tgz fe/meta
  4. BACKUP SNAPSHOT으로 핵심 마트 백업 (OPS-4).
  5. 새 버전 바이너리를 모든 PM에 미리 배포 (Ansible).
  6. 알람 일시 무음(maintenance window) — 단, 절대 끄지 말 것.

순서 — Observer → Follower → Leader → BE

graph LR O["Observer (있으면)"] --> F["Follower"] --> L["Leader"] --> B["BE 1대씩"]

FE 업그레이드 (1대씩)

  1. 대상 FE에서 stop_fe.sh
  2. 새 버전 디렉터리로 심볼릭 링크 변경 (또는 새 위치로 unpack)
  3. start_fe.sh --daemon
  4. SHOW FRONTENDS에서 alive=true + version이 새 버전인지 확인
  5. 다음 FE로 진행. Leader는 가장 마지막
-- 진행 중 상시 모니터링
SHOW FRONTENDS;
SHOW BACKENDS;
SELECT count(*) FROM information_schema.tablet_schedules WHERE state IN ('RUNNING','PENDING');

BE 업그레이드 (1~2대씩)

  1. 대상 BE의 Tablet 복제(보통 3) 중 하나가 잠시 빠지므로, 한 번에 1~2대만.
  2. stop_be.sh
  3. 새 버전 위치로 교체
  4. start_be.sh --daemon
  5. SHOW BACKENDS에서 alive·version 확인 + clone queue 비워질 때까지 대기
  6. 다음 BE로

Ansible 자동화 예

# roles/upgrade_be/tasks/main.yml
- name: stop be
  systemd: name=starrocks-be state=stopped

- name: switch symlink to new version
  file: src=/opt/StarRocks-{{ new_version }}/be path=/opt/sr/be state=link force=yes

- name: start be
  systemd: name=starrocks-be state=started

- name: wait alive
  shell: |
    until mysql -h fe1 -P 9030 -uroot -p$ROOT_PWD -N -B -e \
      "SELECT count(*) FROM information_schema.backends WHERE IP='{{ ansible_default_ipv4.address }}' AND alive='true'" \
      | grep -q '^1$'; do sleep 5; done

- name: wait clone queue empty
  shell: |
    until [ "$(mysql -h fe1 -P 9030 -uroot -p$ROOT_PWD -N -B -e \
      'SELECT count(*) FROM information_schema.tablet_schedules WHERE state IN (\"RUNNING\",\"PENDING\")')" -lt 5 ]; do sleep 10; done

# 그리고 serial: 1 (또는 2) 로 호스트별 순차 실행
- hosts: be
  serial: 1
  roles: [upgrade_be]

사후 점검

  • SHOW FRONTENDS / BACKENDS — version 일치 확인
  • 주요 대시보드 SQL 응답 — 회귀 없는지
  • Routine Load 상태 RUNNING
  • MV 상태 ACTIVE + last_refresh SUCCESS
  • p99 query latency 변화 추적 (Grafana)

롤백 (긴급)

  1. 업그레이드 후 치명적 회귀 발생 시.
  2. BE는 같은 절차의 역순으로 이전 버전 심볼릭 링크로 교체.
  3. FE는 이전 버전이 같은 메이저 라인이어야 안전. 메이저 다운그레이드는 메타 호환 문제 발생 가능 — 가능하면 피한다.
  4. 최악의 경우 FE 메타 백업으로 복구 후 이전 버전 재기동.
클러스터 적용 노트
  • 약 17 BE를 1~2대씩 → 약 8~17 단계. 한 단계당 5~10분이라 새벽 2~6시 윈도면 충분.
  • 메이저 점프(3.x → 4.x)는 LTS를 거쳐 단계적으로.
  • 장애 시뮬레이션: 분기 1회 BE 1대 강제 종료 → 자동 복구 확인이 좋은 훈련.
