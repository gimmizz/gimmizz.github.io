GitOps — StarRocks 클러스터·스키마를 Git으로

왜

  • 클러스터 정의·스키마·MV 정의가 Git에 있으면 리뷰·롤백·감사가 자동 보장.
  • 운영자별로 직접 ALTER하던 패턴 → PR 기반 변경.
  • K8s Operator + ArgoCD/Flux 조합이 가장 자연스럽다.

GitOps에 두는 것 vs 안 두는 것

구분예시
✅ Git에 둠Operator/StarRocksCluster CR, ConfigMap(be.conf/fe.conf), CREATE TABLE/MV DDL, Routine Load 정의, 권한·역할 정의
✅ Git에 둠 (별도 dbt repo)마트 모델 SQL, MV, 변환 로직
❌ Git에 안 둠비밀번호/keytab(Vault), 런타임 메타(BDB JE), 사용자 데이터, audit log

ArgoCD 패턴

# Application 정의 (ArgoCD)
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: starrocks-prod
  namespace: argocd
spec:
  project: data-platform
  source:
    repoURL: https://git.internal/sr-prod
    path: k8s/
    targetRevision: main
  destination:
    server: https://kubernetes.default.svc
    namespace: starrocks
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true

DDL 적용 패턴

K8s Operator는 클러스터 인프라까지만 책임. 스키마·MV·권한은 별도 잡으로 적용.

# GitHub Actions / Argo Workflow / Jenkins 어느 쪽이든
# PR merge → 새 SQL 파일 (예: 0050_*.sql)이 추가되면 적용

- name: apply DDL
  run: |
    for f in $(git diff --name-only HEAD~1 HEAD | grep '^sql/.*\.sql$' | sort); do
      mysql -h sr-prod-mysql-ext -P 9030 -u ddl_user -p$DDL_PWD < "$f"
    done

도구별 대안: Liquibase 또는 Flyway가 변경 추적 + 멱등성 + 롤백을 잘 다룸.

dbt 모델은 별도 흐름

dbt는 자체 상태 추적이 있어 GitOps와 자연스럽게 결합. dbt repo는 별도이거나 같은 repo의 dbt/ 하위로.

롤백

  • K8s 인프라: git revert → ArgoCD 자동 동기.
  • DDL: 이전 SQL의 reverse를 새 PR로 (DROP COLUMN, REVERT INDEX 등). DDL은 단순 git revert가 위험.

감사 로그

SR audit_loader 플러그인 + Git PR 기록을 함께 보관하면 "누가 언제 무엇을 바꿨는지" 추적 완성.

클러스터 적용 노트
  • 온프레 PM 환경에서도 클러스터 정의를 Ansible repo + DDL을 별도 git repo에 두는 식으로 GitOps 사상의 일부를 가져올 수 있음.
  • 처음부터 100% GitOps는 부담 — 단계적으로: ① DDL을 git repo에 모음 → ② dbt 도입 → ③ K8s 이행 시 ArgoCD.
