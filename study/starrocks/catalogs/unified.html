6-5. Unified Catalog (v3.2+)

📌 한 줄 요약

한 메타스토어에 Hive parquet, Iceberg, Hudi, Delta 테이블이 섞여 있을 때 카탈로그 4개를 따로 만들지 않고 하나로 묶어서 다룰 수 있다. StarRocks가 테이블 형식을 자동 인식.

기본 PROPERTIES

CREATE EXTERNAL CATALOG unified
PROPERTIES (
  "type"                   = "unified",
  "unified.metastore.type" = "hive",        -- "hive" | "glue"
  "hive.metastore.uris"    = "thrift://hms.example.com:9083",
  "aws.s3.endpoint"        = "https://minio.internal:9000",
  "aws.s3.access_key"      = "...",
  "aws.s3.secret_key"      = "...",
  "aws.s3.enable_path_style_access" = "true"
);

동작 방식

graph LR SQL["SELECT ... FROM unified.db.tbl"] FE["FE — 메타 분석"] HMS["HMS / Glue"] H["Hive Reader"] I["Iceberg Reader"] HU["Hudi Reader"] D["Delta Reader"] SQL --> FE --> HMS FE -- "tbl이 Iceberg 형식이면" --> I FE -- "tbl이 Hudi 형식이면" --> HU FE -- "tbl이 Delta 형식이면" --> D FE -- "그 외" --> H

그림 6-5-1. Unified Catalog의 자동 포맷 라우팅

주의사항

  • 모든 테이블이 같은 인증/스토리지 자격을 공유 — 권한 분리가 필요한 경우 별도 카탈로그가 더 적합.
  • 형식별 고급 기능(Iceberg Time Travel, Hudi Incremental Query 등)은 그대로 사용 가능.
  • HMS의 테이블 프로퍼티(`table_type=ICEBERG` 등)에 의존해 형식을 결정하므로 메타 정합성이 중요.
클러스터 적용 노트
  • HMS 안에 다양한 포맷이 섞여 있고, BI/분석가에게 카탈로그 이름을 단순하게 노출하고 싶다면 1순위.
  • 다만 권한·보안 정책이 달라야 하는 데이터(예: PII 테이블)는 별도 카탈로그로 분리하는 게 깔끔.
