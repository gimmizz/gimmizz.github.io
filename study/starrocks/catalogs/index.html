6. 카탈로그 개요 — Internal · External · Unified

📌 한 줄 요약

StarRocks의 카탈로그는 "데이터 소스 묶음"이다. 자체 테이블(default_catalog)과 외부 시스템(Hive, Iceberg, MySQL...)을 같은 SQL 안에서 다룰 수 있게 해 준다. v3.2부터는 데이터 레이크 카탈로그들을 통합한 Unified Catalog도 가능.

graph LR SR["StarRocks SQL"] SR --> DC["default_catalog
(내부 OLAP 테이블)"] SR --> H["Hive Catalog"] SR --> I["Iceberg / Hudi / Delta"] SR --> J["JDBC Catalog"] SR --> P["Paimon / ES / Kudu / ..."] H --> HMS["HMS / Glue"] --> S3["HDFS · S3 · OSS · MinIO"] I --> RC["REST/HMS/Glue"] --> S3 J --> RDB["MySQL · PG · Oracle · ClickHouse"]

그림 6-1. 한 StarRocks 인스턴스에서 다수 카탈로그 결합 SQL

6.1 카탈로그 개념

MySQL의 "데이터베이스"보다 한 단계 위에 있는 개념이라 생각하면 됩니다. StarRocks는 다음과 같이 3단으로 객체를 식별합니다. 

catalog.database.table

예) default_catalog.mart.orders        -- 내부 테이블
    hive_prod.dwh.events               -- Hive 카탈로그의 테이블
    mysql_ops.app.users                -- JDBC 카탈로그의 테이블

6.2 3가지 카탈로그 종류

Internal Catalog (default_catalog)

  • StarRocks 내부에 저장되는 테이블 전체.
  • 클러스터당 단 하나, 이름 고정 (default_catalog).
  • CREATE DATABASE·CREATE TABLE로 만든 객체가 모두 이 카탈로그 안에 들어감.

External Catalog

외부 메타스토어/시스템을 마운트해서 직접 쿼리하는 카탈로그입니다. 데이터 이동이 없습니다.

카탈로그 종류대상 시스템지원 시점
HiveHMS·AWS Glue로 관리되는 Hive 테이블 (HDFS/S3)v2.3+
IcebergApache Iceberg 테이블v2.4+
HudiApache Hudi 테이블 (CoW/MoR)v2.4+
Delta LakeDelta Lake 테이블v2.5+
JDBCMySQL / PostgreSQL / Oracle 등 JDBC 호환 RDBv3.0+
ElasticsearchElasticsearch 인덱스v3.1+
PaimonApache Paimon 테이블v3.1+
MaxComputeAlibaba MaxComputev3.x+
KuduApache Kuduv3.x+

Unified Catalog (v3.2+)

Hive·Iceberg·Hudi·Delta를 한 카탈로그로 묶어서 관리합니다. 같은 HMS 안에 형식이 섞여 있는 환경(예: 일부 테이블은 Hive parquet, 일부는 Iceberg)에서 굉장히 유용합니다. 별도의 catalog 4개를 만들 필요 없이 하나로 모든 형식을 자동 인식합니다.

6.3 사용법 기본기

-- 카탈로그 목록
SHOW CATALOGS;

-- 현재 세션의 카탈로그 전환
SET CATALOG hive_prod;
SHOW DATABASES;
USE dwh;
SELECT * FROM events LIMIT 10;

-- 전환 없이 풀 네임 사용
SELECT count(*)
FROM hive_prod.dwh.events e
JOIN default_catalog.mart.user_dim u
  ON e.user_id = u.user_id
WHERE e.dt = '2026-05-07';

6.4 외부 카탈로그가 동작하는 구조

외부 카탈로그를 쓰려면 두 종류의 시스템에 접근할 수 있어야 합니다.

  1. 메타스토어 — 어떤 테이블/파티션이 있는지 알려주는 카탈로그. (Hive Metastore, AWS Glue, REST/Iceberg Catalog, JDBC URL ...)
  2. 스토리지 — 실제 파일이 놓여 있는 곳. (HDFS, S3, OSS, Azure Blob, MinIO, NFS ...)

FE가 메타스토어에서 메타데이터를 받고, BE가 스토리지에서 실제 데이터를 병렬로 읽습니다. 그래서 BE 모두에서 스토리지(HDFS/S3 등)에 접근 가능해야 합니다.

6.5 인증 방식

외부 카탈로그를 만들 때 PROPERTIES로 인증 정보를 넣습니다. 대표적인 패턴.

HMS — Thrift

"hive.metastore.type" = "hive",
"hive.metastore.uris" = "thrift://hms1:9083,thrift://hms2:9083"

HDFS — Simple/Kerberos

# Simple
"hadoop.security.authentication" = "simple",
"username" = "sr"

# Kerberos
"hadoop.security.authentication" = "kerberos",
"kerberos_principal" = "sr/_HOST@REALM",
"kerberos_keytab" = "/etc/security/keytabs/sr.keytab",
"hadoop.kerberos.principal.pattern" = "..."

S3 호환

"aws.s3.access_key" = "...",
"aws.s3.secret_key" = "...",
"aws.s3.region" = "ap-northeast-2",
"aws.s3.endpoint" = "https://s3.ap-northeast-2.amazonaws.com"

6.6 권한 모델 (RBAC)

-- 카탈로그 사용 권한
GRANT USAGE ON CATALOG hive_prod TO ROLE bi_reader;

-- 카탈로그 전체 SELECT
GRANT SELECT ON ALL TABLES IN ALL DATABASES IN CATALOG hive_prod TO ROLE bi_reader;

-- 특정 DB만
GRANT SELECT ON ALL TABLES IN DATABASE hive_prod.dwh TO ROLE bi_reader;

6.7 메타데이터 캐시

FE는 외부 카탈로그 메타데이터를 캐시합니다 — 매번 HMS를 호출하지 않게 하기 위함. 필요할 때 수동 갱신: 

-- 카탈로그 단위
REFRESH EXTERNAL TABLE hive_prod.dwh.events;
-- 또는
REFRESH CATALOG hive_prod;

파티션이 새로 추가되었는데도 보이지 않는다면 거의 메타 캐시 문제입니다. 자동 새로고침 주기는 fe.conf의 hive_meta_cache_refresh_interval_s 등으로 조정할 수 있습니다.

6.8 Data Cache (외부 데이터 가속)

외부 카탈로그를 매번 원격에서 읽으면 느릴 수 있습니다. StarRocks는 BE 로컬 디스크에 외부 파일 일부를 캐시하는 Data Cache를 제공합니다(v2.x+, v3.x에서 강화). 

# be.conf
datacache_enable = true
datacache_disk_size = "100G"
datacache_disk_path = "/data/sr/cache"

자주 읽히는 핫 파티션은 거의 로컬 SSD에서 읽히게 되어 외부 카탈로그 쿼리가 내부 테이블에 가깝게 빨라집니다.

  • Data Cache는 처음부터 켜자: SSD 8개 중 하나(또는 일부)를 외부 카탈로그 캐시 전용으로 떼어두면 Hive/Iceberg 쿼리가 내부 테이블 수준으로 빨라집니다. 예) /data/sr/disk8을 datacache_disk_path로.
  • HMS·HDFS·MySQL 접속 정보는 별도 비밀 저장소에서 주입: fe.conf나 SQL에 평문 비밀번호를 박지 말고, 환경 변수/Vault에서 주입하는 패턴 권장.
  • Unified Catalog 우선 검토: HMS 안에 Hive·Iceberg·Hudi가 섞여 있을 가능성이 있다면 처음부터 Unified로 묶는 편이 운영 단순. 단, 권한·보안 정책이 통합되어야 합니다.
  • JDBC Catalog는 "쿼리만": MySQL을 JDBC Catalog로 붙여 OLAP 쿼리에 합류시킬 수 있지만, 무거운 집계는 원천 MySQL을 죽일 수 있습니다. 자주 쓰는 결합은 Async MV로 미리 떠 두고, MV가 자동 사용되게 하는 패턴이 안전.
  • 네트워크: HMS·HDFS NameNode·DataNode·MySQL 모두 BE 약 17대에서 직접 접속 가능해야 함 — 사전에 방화벽 일괄 허용 필요.
