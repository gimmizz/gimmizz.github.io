6-2. JDBC Catalog (MySQL · PostgreSQL · Oracle ...)

📌 한 줄 요약

JDBC URI + 드라이버 JAR + 사용자 정보만 주면 RDB를 카탈로그처럼 마운트할 수 있다. 가벼운 조회·통계 결합에 좋고, 무거운 집계는 원천 RDB에 부담을 주므로 Async MV로 옮기는 게 안전하다.

6-2.1 지원 데이터베이스

RDB지원 시점참고
MySQLv3.0+가장 안정적·범용
PostgreSQLv3.0+—
Oraclev3.2.9+NUMBER/TIMESTAMP 매핑 옵션 존재
SQL Serverv3.3.1+—
ClickHousev3.3.0+ (실험적)—

6-2.2 MySQL 카탈로그 예시

CREATE EXTERNAL CATALOG mysql_ops
PROPERTIES (
  "type"          = "jdbc",
  "user"          = "sr_reader",
  "password"      = "...",
  "jdbc_uri"      = "jdbc:mysql://mysql.internal:3306",
  "driver_url"    = "file:///opt/sr/drivers/mysql-connector-j-8.4.0.jar",
  "driver_class"  = "com.mysql.cj.jdbc.Driver"
);

-- 사용
SET CATALOG mysql_ops;
SHOW DATABASES;
USE app;
SELECT count(*) FROM users;

-- 풀 네임 + 다른 카탈로그와 조인
SELECT u.id, u.name, h.event_type
FROM mysql_ops.app.users u
JOIN hive_prod.dwh.events h
  ON u.id = h.user_id
WHERE h.dt = '2026-05-07';

6-2.3 드라이버 배치 방법

driver_url은 세 가지 방식이 가능합니다.

  1. HTTP/HTTPS URL: Maven Central 등에서 원격으로 다운로드.
    예: https://repo1.maven.org/maven2/com/mysql/mysql-connector-j/8.4.0/mysql-connector-j-8.4.0.jar
  2. file:// (FE/BE 모두 같은 경로): 인터넷이 막힌 사내망에서 가장 안정적. 모든 FE·BE 노드의 동일 경로(예: /opt/sr/drivers/...)에 JAR을 미리 깔아둡니다.
  3. file:// (NFS 등 공유 경로): 모든 노드가 같은 마운트로 접근 가능하다면 사용 가능.

FE는 카탈로그 생성 시점에 JAR을 받고, BE는 첫 쿼리 시점에 받습니다 — 따라서 두 종류 노드 모두 다운로드 가능해야 합니다.

⚠️ Java 환경

BE의 start_be.sh에서 JAVA_HOME이 JRE가 아니라 JDK를 가리켜야 합니다 (JNI 호환).

6-2.4 Oracle 카탈로그 추가 옵션

"oracle.number.default-scale"     = "6",
"oracle.temporal.to-datetime"     = "false",
"oracle.timestamptz.to-datetime"  = "false"

Oracle의 NUMBER(p)는 scale이 정해져 있지 않을 수 있어, 변환 기본 scale을 결정합니다. Temporal/TimezoneTimestamp 옵션은 StarRocks에서 DATETIME으로 변환할지 VARCHAR(64)로 변환할지 결정합니다.

6-2.5 푸시다운 (Predicate Pushdown)

StarRocks는 SELECT 컬럼 프로젝션과 단순 WHERE 조건을 원천 RDB로 푸시다운하려고 시도합니다. 그러나 모든 조건이 푸시다운되지는 않습니다 (함수, 복잡 표현식, 조인 등). 푸시다운 여부는 EXPLAIN으로 확인할 수 있습니다. 

EXPLAIN
SELECT id, name FROM mysql_ops.app.users WHERE id BETWEEN 100 AND 200;

플랜에 JDBC_SCAN_NODE가 보이고, 그 아래 jdbc-query로 실제 RDB에 보낼 SQL이 표시됩니다.

6-2.6 INSERT 지원

JDBC Catalog는 INSERT INTO ... SELECT로 데이터를 원천 RDB에 쓰는 것도 지원합니다.

INSERT INTO mysql_ops.app.user_score
SELECT user_id, sum(amount) AS score
FROM hive_prod.dwh.events
WHERE dt = '2026-05-07'
GROUP BY user_id;

다만 RDB가 OLTP라면 한 번에 너무 큰 양을 넣지 않도록 분할 적재해야 합니다.

6-2.7 한계와 자주 만나는 함정

  • RDB 부하: 큰 테이블에 GROUP BY를 그대로 던지면 원천 RDB가 죽을 수 있음. 운영 RDB는 가능하면 읽기 복제본(MySQL replica 등)에 붙일 것.
  • 드라이버 누락: BE 한 대라도 JAR 다운로드 실패 시 첫 쿼리에서 에러. 사내망에선 file:// 방식 권장.
  • 네트워크: 모든 BE가 RDB 포트(예: 3306)에 접근 가능해야 함.
  • 인증 정보 노출: SHOW CREATE CATALOG로 비밀번호가 노출될 수 있음 — RBAC로 카탈로그 메타 권한을 제한.
  • 스키마 변경: RDB 컬럼이 바뀌면 REFRESH로 메타 갱신.
클러스터 적용 노트 — MySQL 등 RDB를 안전하게 붙이기
  • 읽기 전용 계정 + read replica 권장: 운영 MySQL primary가 아닌 read replica를 카탈로그 대상으로. sr_reader 같은 전용 계정에 SELECT만.
  • 드라이버는 /opt/sr/drivers/에 통일: 약 20대 PM에 Ansible로 같은 JAR을 배포. 인터넷 의존을 끊습니다.
  • Async MV로 캐시: 같은 RDB 테이블을 자주 조회한다면 Async MV로 떠두고, MV 자동 사용을 켭니다 (7장 쿼리/MV 참고). RDB는 30분~1시간 주기로만 건드리고, 분석가는 MV에서 응답을 받습니다.
  • 여러 RDB를 묶을 때: mysql_ops_a, mysql_ops_b처럼 카탈로그 이름에 환경/소스 구분을 넣으면 권한 관리가 깔끔합니다.
