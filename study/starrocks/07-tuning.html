8. 튜닝 가이드 — 약 20대 PM 환경 기준

📌 한 줄 요약

대부분의 튜닝은 (1) BE 메모리 분배, (2) Compaction, (3) MV·Data Cache, (4) 동시성·세션 변수 — 이 4가지에서 끝난다.

그림 8-1. 약 20대 PM 환경의 BE 메모리 분배 권장 (mem_limit=80%)

8.1 BE 메모리 분배

BE의 메모리는 다음과 같이 나뉘어 사용됩니다.

구획역할대표 옵션
전체 한도BE가 쓸 수 있는 OS 메모리의 상한mem_limit (보통 80%)
쿼리 풀실행 중 쿼리들이 사용query_mem_limit(세션) / load_mem_limit
Page Cache컬럼 페이지 캐시storage_page_cache_limit
Data Cache외부 카탈로그 데이터 캐시datacache_mem_size
PK IndexPrimary Key 인덱스enable_persistent_index (테이블 옵션)
Compaction컴팩션 워커compaction_mem_limit
 
# be.conf 예시 (256GB / 노드 가정)
mem_limit                  = 80%       # OS 메모리의 80%
storage_page_cache_limit   = 20%
datacache_mem_size         = 10%
compaction_mem_limit       = 10%

8.2 디스크 — SSD 8개를 모두 활용하기

# be.conf
storage_root_path = /data/sr/disk1,medium:SSD;\
                    /data/sr/disk2,medium:SSD;\
                    /data/sr/disk3,medium:SSD;\
                    /data/sr/disk4,medium:SSD;\
                    /data/sr/disk5,medium:SSD;\
                    /data/sr/disk6,medium:SSD;\
                    /data/sr/disk7,medium:SSD;\
                    /data/sr/disk8,medium:SSD

BE는 8개 디스크에 Tablet을 라운드로빈으로 분산합니다 — 한 디스크가 병목이 되지 않도록. 같은 PM 안에서도 디스크별 사용률을 주기적으로 확인하세요(SHOW PROC '/backends/<id>/disks';). 특정 디스크가 과편향이면 Tablet 마이그레이션이나 디스크 추가/제거를 검토.

8.3 Compaction

적재가 빈번하면 작은 segment 파일이 많아져서 쿼리 시 머지 비용이 커집니다. 컴팩션이 그것을 정리합니다. 가장 자주 보는 지표는 compaction score로, 이 값이 100을 넘으면 적체 신호입니다. 

# be.conf — Compaction 워커 수
base_compaction_num_threads_per_disk        = 1
cumulative_compaction_num_threads_per_disk  = 1
update_compaction_num_threads_per_disk      = 1   # PK 테이블용

# 적체가 큰 환경에서는 디스크당 워커를 늘릴 수 있음
# 단 너무 늘리면 쿼리 IO와 경쟁

8.4 Data Cache (외부 카탈로그용)

# be.conf
datacache_enable    = true
datacache_disk_path = "/data/sr/cache"   # 디스크 1개 전용 권장
datacache_disk_size = "400G"             # 디스크 용량의 90% 정도
datacache_mem_size  = "10%"

8.5 쿼리 동시성 / 병렬도

-- 세션 변수
SET parallel_fragment_exec_instance_num = 8;   -- BE당 fragment 인스턴스 수
SET pipeline_dop = 0;                          -- 0이면 자동 (코어 수 기반)
SET query_mem_limit = 32 * 1024 * 1024 * 1024; -- 32GB

-- 글로벌로 두려면
SET GLOBAL parallel_fragment_exec_instance_num = 8;

pipeline_dop = 0이 기본 권장. 명시적으로 줄이는 건 하나의 무거운 쿼리가 노드 전체를 먹는 걸 막을 때.

8.8 Query Profile 활용

SET enable_profile = true;
SELECT ...; -- 쿼리 실행

SHOW PROFILELIST;
SHOW PROFILE FROM 'profile_id';

Profile은 각 Operator의 시간/처리 행수/메모리를 보여줍니다. 가장 큰 단일 시간이 어디서 나오는지 — Scan, Hash Join Build, Aggregate, Exchange — 를 본 다음 그 부분에 맞는 튜닝을 합니다.

8.9 성능 점검 체크리스트

  1. 통계가 최신인가? SHOW STATS META FOR ...;
  2. EXPLAIN 결과의 cardinality 추정이 실제와 비슷한가?
  3. 의도한 JOIN 전략으로 풀리는가? 셔플 양이 과한가?
  4. Tablet 수가 충분히 많고 디스크 8개에 잘 분산되어 있는가?
  5. Compaction Score가 100을 넘지 않는가?
  6. 외부 카탈로그 쿼리는 Data Cache 히트율이 충분한가?
  7. 같은 결합/집계가 반복된다면 Async MV 후보인가?
  8. 대시보드 트래픽과 ETL이 한 Resource Group에 섞여 있는가?
클러스터 적용 노트 — 약 20대 PM에서의 튜닝 시작점
  • BE 256GB 메모리 분배 예시: mem_limit=80% ≈ 200GB → 그 안에서 query 풀 100GB / page cache 50GB / data cache 30GB / compaction 20GB. 실측을 보고 미세 조정.
  • SSD 8개 활용: 8개 모두 storage_root_path에 등록. 또는 7개 + 1개(Data Cache 전용) 분리. 외부 카탈로그가 무겁다면 후자가 효과적.
  • Compaction 워커: PK 테이블을 많이 쓰면 update_compaction_num_threads_per_disk = 2~3으로 늘려 적체 방지. 단, 쿼리 IO와 경쟁하므로 모니터링 후 조정.
  • 네트워크: 셔플 비용은 결국 NIC 대역폭에 직격. 10GbE면 한 BE당 1GB/s 정도가 한계. 큰 셔플이 자주 발생하면 Bond/25GbE를 검토.
  • RuntimeFilter: 큰 테이블 - 작은 테이블 조인이 많은 워크로드라면 글로벌 RuntimeFilter가 잘 동작합니다 (기본 켜짐).
  • Tablet 수의 안전대: 한 BE당 Tablet이 수만~십수만이 되면 메타 부담이 커집니다. 약 17 BE × 디스크 8 ≒ Tablet이 너무 적게 잡히지 않게 파티션·버킷 수를 적절히. v3.2+의 Auto Bucket이 보통 잘 잡아 줍니다.
  • 관측 우선: 모든 튜닝의 출발점은 Grafana 대시보드(Compaction Score, Mem Used, IO Util, Query QPS)입니다. 추측보다 그래프.
