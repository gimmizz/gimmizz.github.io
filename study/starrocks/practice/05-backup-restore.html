P-5. BACKUP / RESTORE 실습

📌 목표

S3(MinIO)를 백업 저장소로 등록하고, 테이블 1개를 백업·복원해본다.

저장소 등록 (Repository) 
CREATE REPOSITORY backup_s3
WITH BROKER (
  "fs.s3a.access.key" = "minio",
  "fs.s3a.secret.key" = "minio123",
  "fs.s3a.endpoint"   = "http://minio:9000",
  "fs.s3a.path.style.access" = "true"
)
ON LOCATION "s3a://warehouse/backup/sr/";

SHOW REPOSITORIES;
스냅샷 백업 
BACKUP SNAPSHOT shop.snap_20260507
TO backup_s3
ON (orders)
PROPERTIES ("type" = "full");

SHOW BACKUP FROM shop\G
복원 — 새 DB로 
CREATE DATABASE shop_restored;

RESTORE SNAPSHOT shop_restored.snap_20260507
FROM backup_s3
ON (orders)
PROPERTIES (
  "backup_timestamp" = "<위 SHOW BACKUP에서 복사>",
  "replication_num"  = "1"
);

SHOW RESTORE FROM shop_restored\G

USE shop_restored;
SELECT count(*) FROM orders;
실패 시나리오
  • 저장소 권한 부족: SHOW BACKUP의 FailedMsg 확인. MinIO 정책 점검.
  • 버전 불일치: 다른 메이저 버전으로 복원하면 일부 메타가 호환 안 될 수 있음.
  • 외부 카탈로그 데이터: 백업 대상 아님 — 원천에 책임.
클러스터 적용 노트
  • 운영에서는 매일 야간 BACKUP 스케줄 + 30~90일 보관. NAS/HDFS/객체 스토리지 어느 쪽이든 가능.
  • FE 메타 디렉터리(fe/meta)도 함께 tar 백업. BACKUP은 데이터/스키마, fe/meta는 카탈로그·권한 메타까지 포함.
  • 분기에 한 번 RESTORE 리허설 — 안 해보면 실제 사고 때 시간이 너무 걸림.
