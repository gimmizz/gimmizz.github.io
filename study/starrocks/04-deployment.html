4. 배포 / 클러스터 구축

📌 한 줄 요약

새 클러스터는 보통 FE 3대 + BE ≥3대로 시작한다. 외부 의존이 없으니 절차는 단순하다 — 바이너리 다운로드 → fe.conf/be.conf 수정 → 시작 → SQL로 BE 등록.

4.1 6단계 배포 절차 (공식 가이드)

  1. 요구사항 확인 — CPU/메모리/디스크/네트워크/OS.
  2. 클러스터 크기 설계 — FE/BE(/CN) 노드 수.
  3. 환경 점검 — JDK, ulimit, swap, OS 설정.
  4. 설치 파일 준비 — 바이너리(x86/ARM) / Docker / Kubernetes(Operator·Helm).
  5. StarRocks 시작 — FE 부트스트랩 → BE 등록.
  6. 배포 후 보안 설정 — root 비밀번호, 파라미터 튜닝.

4.2 하드웨어 권장사항 (공식)

노드CPU메모리디스크비고
FE 8 코어 이상 16 GB 이상 100 GB (메타데이터) HDD도 가능. SSD면 더 좋음.
BE 16 코어 이상 64 GB 이상 워크로드에 따라 SSD 강권장. 다중 디스크 가능.
CN BE와 동일 BE와 동일 로컬은 캐시용 실데이터는 객체 스토리지/HDFS에 저장.

BE 디스크 용량 산식은 총 저장소 = 원본 데이터 × 복제본 수 ÷ 압축률 입니다. 기본 복제본은 3, 압축률은 컬럼/포맷에 따라 3:1 ~ 5:1 정도가 일반적입니다(zlib·zstd·LZ4·Snappy).

4.3 OS·런타임 사전 점검

모든 노드에서 미리 확인해야 할 항목.

  • OS: CentOS 7+ / RHEL 8+ / Ubuntu 20.04+ 등 일반 리눅스.
  • JDK: FE는 JDK 11/17 권장(버전에 따라 다름; 4.x는 JDK 17 권장).
  • 커널 파라미터:
    • vm.swappiness=0 (또는 swap off) — OLAP에서 swap은 치명적
    • vm.overcommit_memory=1
    • vm.max_map_count ≥ 262144
    • fs.file-max 충분히 크게
  • ulimit: nofile 65535, nproc 큰 값.
  • THP(Transparent Huge Pages): 일반적으로 비활성 권장(never).
  • NTP: FE 간 시간 동기 필수 (chrony/ntpd).
  • 호스트네임/방화벽: FE↔FE, FE↔BE, BE↔BE 통신 포트 개방.

4.4 디렉터리 레이아웃

StarRocks-x.y.z/
├── fe/
│   ├── bin/          # start_fe.sh, stop_fe.sh
│   ├── conf/         # fe.conf
│   ├── lib/
│   ├── log/          # 로그
│   └── meta/         # BDB JE 메타데이터 (중요!)
└── be/
    ├── bin/          # start_be.sh, stop_be.sh
    ├── conf/         # be.conf
    ├── lib/
    ├── log/
    └── storage/      # 또는 storage_root_path로 외부 디스크들 지정

4.5 FE 설정 핵심 (fe.conf)

# 메타데이터 디렉터리 (반드시 백업 가능한 안정적 디스크)
meta_dir = /data/sr/fe/meta

# 클러스터 내 자기 IP (호스트네임이 정확히 잡혀 있다면 priority_networks 권장)
priority_networks = 10.0.0.0/24

# 포트 (기본값 그대로 두는 게 일반적)
http_port = 8030
rpc_port = 9020
query_port = 9030
edit_log_port = 9010

# 메모리 — JVM 옵션은 fe.conf의 JAVA_OPTS_FOR_JDK_X 에서 -Xmx 지정
# 8GB ~ 32GB 사이가 보통. 메타데이터/연결 수에 비례.

4.6 BE 설정 핵심 (be.conf)

# 자기 IP (priority_networks 권장)
priority_networks = 10.0.0.0/24

# 데이터 디렉터리 — 디스크별로 ; 로 구분
storage_root_path = /data/sr/disk1;/data/sr/disk2;/data/sr/disk3;/data/sr/disk4;/data/sr/disk5;/data/sr/disk6;/data/sr/disk7;/data/sr/disk8

# 매체 명시 (SSD/HDD)
# storage_root_path = /data/sr/disk1,medium:SSD;/data/sr/disk2,medium:SSD;...

# 메모리 한도 — 노드 메모리의 80% 정도가 일반적
mem_limit = 80%

# 동시 쿼리 동시성
be_http_port = 8040
heartbeat_service_port = 9050
brpc_port = 8060
be_port = 9060

4.7 부트스트랩 절차

  1. FE Leader 첫 기동: 
    cd /opt/StarRocks-x.y.z/fe
bin/start_fe.sh --daemon
  2. MySQL CLI로 접속: 
    mysql -h fe1 -P 9030 -u root
  3. 다른 FE를 Follower로 추가: 
    ALTER SYSTEM ADD FOLLOWER "fe2:9010";
ALTER SYSTEM ADD FOLLOWER "fe3:9010";
  4. 각 Follower 노드에서 시작 (이때 --helper fe1:9010 옵션): 
    bin/start_fe.sh --daemon --helper fe1:9010
  5. BE 등록: 
    ALTER SYSTEM ADD BACKEND "be1:9050";
ALTER SYSTEM ADD BACKEND "be2:9050";
... (모든 BE 반복)
  6. 각 BE 노드에서 시작: 
    cd /opt/StarRocks-x.y.z/be
bin/start_be.sh --daemon
  7. 상태 확인: 
    SHOW FRONTENDS;
SHOW BACKENDS;

4.8 배포 후 보안·튜닝

  • root 비밀번호 설정: SET PASSWORD FOR 'root' = PASSWORD('....');
  • 전용 계정 생성: CREATE USER 'app' IDENTIFIED BY '...'; GRANT SELECT ON *.* TO 'app';
  • FQDN/HOSTNAME 모드: 고정 IP가 어려운 환경(K8s 등)이라면 FQDN 모드를 검토.
  • 모니터링: Prometheus + Grafana 대시보드(공식 제공).

4.9 설치 방식 비교

방식장점단점추천 케이스
바이너리 수동가장 단순, 모든 옵션 직접스크립트화 필요온프레 PM 클러스터 (우리 케이스)
Docker로컬 테스트에 빠름프로덕션 권장 X개발/학습
K8s Operator탄력적·자동복구K8s 운영 노하우 필요퍼블릭 클라우드 + Shared-data
HelmOperator의 패키지화위와 동일K8s 표준 팀
클러스터 적용 노트 — 약 20대 PM 권장 토폴로지

전제: 노드당 메모리 16GB×16 = 256GB, SSD 480GB×8 = 약 3.84TB, 약 20대.

  • FE 3대 — 별도 PM 3대를 FE 전용으로. 디스크는 PM의 시스템 디스크 일부에 100~500GB만 할당하면 충분(fe/meta). JDK 17 + Xmx 32~64GB 정도.
  • BE 17대 — 나머지 모두 BE. storage_root_path에 8개 SSD 경로 등록. mem_limit = 80% ≒ 200GB. 3복제로 가용 데이터 ~22TB.
  • BI 트래픽이 매우 많은 경우에만 Observer 1~2대 추가 — 평소엔 불필요.
  • NTP, 방화벽, 호스트네임은 OS 이미지 단계에서 통일 — 사후 수정은 골치 아픕니다.
  • JDK는 4.x 라인 사용 시 JDK 17 통일.

스크립트 자동화 팁: bash/Ansible로 다음을 템플릿화하면 약 20대 정도는 30분 이내에 배포 가능합니다.

  1. StarRocks 바이너리 배포(같은 위치에 unpack)
  2. fe.conf/be.conf 템플릿 + 노드별 IP 주입
  3. 각 노드 8개 디스크 마운트 확인 (/data/sr/disk1..8)
  4. FE Leader → Follower 추가 → BE 일괄 ADD
  5. SHOW BACKENDS 결과 alive 갯수 확인

Shared-data를 검토할 시점: 사내에 안정적으로 사용 가능한 S3 호환 객체 스토리지(MinIO 등)가 있고, 탄력적으로 컴퓨팅을 늘렸다 줄였다 해야 할 때. 지금은 우선 Shared-nothing으로 단순하게 가는 게 유리합니다.
