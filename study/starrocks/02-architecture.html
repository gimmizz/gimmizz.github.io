2. 아키텍처 — FE / BE / CN과 두 가지 모드

📌 한 줄 요약

StarRocks는 FE(메타데이터·옵티마이저)와 BE(스토리지+실행) 두 종류 노드만 있다. 외부 의존(ZK·HMS·etcd 등) 없이도 동작한다. 스토리지 분리가 필요하면 BE 자리에 stateless인 CN을 두고 객체 스토리지를 붙이는 Shared-data 모드를 쓴다.

2.1 전체 그림

StarRocks 클러스터는 두 종류의 노드만으로 구성됩니다. 

             ┌──────────────────────────────┐
   클라이언트  │ MySQL CLI · JDBC · BI 도구    │
             └──────────────┬───────────────┘
                            │ MySQL 프로토콜 (port 9030)
                  ┌─────────┴─────────┐
                  │       FE (3대)    │  ← 메타데이터, 옵티마이저
                  │  Leader/Follower/ │  ← Raft(BDB JE) 동기화
                  │       Observer    │
                  └─────────┬─────────┘
                            │ Thrift (port 9020) 등
        ┌───────────┬───────┴────────┬────────────┐
        │           │                │            │
   ┌────▼──┐   ┌────▼──┐        ┌────▼──┐    ┌────▼──┐
   │ BE 1  │   │ BE 2  │   …    │ BE N  │    │ ...   │   ← 데이터 저장 + 쿼리 실행
   │ Local │   │ Local │        │ Local │    │       │
   │ SSDx8 │   │ SSDx8 │        │ SSDx8 │    │       │
   └───────┘   └───────┘        └───────┘    └───────┘

외부 의존성이 거의 없습니다. ZooKeeper도, etcd도, 별도 메타스토어도 필요 없습니다 — FE에 BDB JE가 내장되어 있기 때문입니다. 이 단순함이 운영 측면에서 큰 이점입니다.

2.2 FE (Frontend) 노드

FE의 책임은 다음 네 가지입니다.

  1. 메타데이터 관리 — 테이블, 컬럼, 파티션, Tablet 위치, 권한 등을 보관.
  2. 클라이언트 연결 — MySQL 프로토콜로 클라이언트 수용.
  3. SQL 파싱 + 옵티마이저(CBO) — 논리 플랜 → 물리 플랜.
  4. 스케줄링 — 만들어진 플랜 단편을 BE에 분배·조정.

FE 역할 — Leader / Follower / Observer

역할읽기쓰기리더 선출 참여특징
LeaderOOO메타데이터 쓰기를 단독 수행. 변경사항을 BDB JE Raft로 다른 FE에 복제.
FollowerOXOLeader 후보. Leader 장애 시 자동 승격.
ObserverOXX읽기 분산용. 선출에 끼지 않으므로 다수 추가해도 클러스터 부담이 작음.

Follower 과반(quorum)이 살아 있어야 메타데이터 쓰기가 성공합니다. 따라서 FE는 거의 항상 홀수(3, 5, 7)로 둡니다. 메타데이터 변경은 Follower 과반 동기화가 끝난 다음에야 클라이언트에 success가 반환됩니다.

FE의 메타데이터 저장

FE는 메타데이터를 메모리에 캐시하면서 동시에 디스크의 BDB JE에 영속합니다. 모든 FE가 메타데이터의 완전 사본을 들고 있다는 점이 중요합니다 — 어느 FE를 읽어도 같은 결과가 나오고, Leader가 죽으면 Follower 중 하나가 즉시 Leader가 됩니다.

  • FE 메타 디렉터리: fe/meta
  • FE 로그 디렉터리: fe/log

2.3 BE (Backend) 노드

BE는 두 가지 일을 동시에 합니다.

  1. 데이터 저장 — 로컬 SSD에 컬럼 형식으로 저장하고, 인덱스(Bitmap, Bloom Filter, Zone Map 등)를 함께 만듦.
  2. 쿼리 실행 — FE에서 받은 물리 플랜 단편을 벡터화 엔진으로 처리.

이 "데이터를 가진 노드가 그 데이터를 직접 처리한다(data locality)" 원칙 덕분에 Shared-nothing 모드에서는 네트워크 셔플이 최소화되어 매우 빠르게 동작합니다.

BE의 디스크 구성

BE 설정 파일 be/conf/be.conf의 storage_root_path에 디스크 경로를 콤마로 나열합니다. 

# 예시: 8개 SSD를 모두 BE 스토리지로 사용
storage_root_path = /data/sr/disk1;/data/sr/disk2;/data/sr/disk3;/data/sr/disk4;/data/sr/disk5;/data/sr/disk6;/data/sr/disk7;/data/sr/disk8

BE는 Tablet을 디스크들에 자동으로 분산 배치합니다. 디스크당 매체(SSD/HDD)도 함께 명시할 수 있습니다(;medium=ssd). 매체 정보는 핫/콜드 데이터 분리(Storage Tier)에서 사용됩니다.

Tablet — 데이터 분배 단위

StarRocks 테이블은 다음과 같이 분할됩니다. 

테이블
 └─ 파티션 (예: 일자별)
     └─ 버킷 = Tablet (해시 분배)
         └─ 복제본 N개 (보통 3)

복제본은 서로 다른 BE에 배치되어 단일 BE 장애에서도 쿼리가 가능합니다. 클러스터 리밸런스, 복구, 마이그레이션 모두 Tablet 단위로 일어납니다.

2.4 CN (Compute Node)

CN은 BE에서 스토리지 책임만 떼어낸 stateless 노드입니다. 데이터는 외부 객체 스토리지(S3, OSS, MinIO, Azure Blob, GCS, HDFS)에 두고, CN은 로컬 디스크/메모리에 캐시만 보관합니다.

  • 몇 초 내에 추가/제거 가능 (데이터 리밸런스 불필요).
  • 비용 효율적 — 객체 스토리지가 일반적으로 SSD보다 GB당 저렴.
  • 탄력적 — 야간/낮 시간대에 따라 CN 수를 늘렸다 줄였다 가능.

단, External Catalog를 쓰는 워크로드라면 BE에서도 데이터 레이크 직접 쿼리가 가능하므로, CN은 주로 StarRocks 자체 데이터를 객체 스토리지에 두는 Shared-data 모드에 사용된다고 이해하면 됩니다.

2.5 두 가지 배포 모드 — Shared-Nothing vs Shared-Data

구분Shared-Nothing (기본)Shared-Data
노드 구성FE + BEFE + CN + 객체 스토리지/HDFS
데이터 위치BE 로컬 SSD객체 스토리지 (CN은 캐시만)
복제Tablet N복제 (보통 3)객체 스토리지의 내구성에 위임 (단일 사본)
확장BE 추가 시 Tablet 리밸런스 필요CN 추가/제거 즉시 효과
비용SSD 사용량에 비례객체 스토리지 + 일부 캐시 SSD
최적 워크로드저지연 쿼리, 고동시성탄력적 스케일, 비용 절감, 데이터 공유

Shared-Data 모드의 캐시 계층

Shared-data 모드에서는 데이터가 원격(객체 스토리지)에 있으므로 매번 가져오면 느립니다. CN은 메모리 → 로컬 디스크 → 원격의 3단 계층 캐시를 운용합니다.

  • 핫 데이터는 캐시에서 직접 스캔.
  • 콜드 데이터는 원격에서 읽으며 동시에 캐시에 기록(Prefetch + 채워넣기).
  • 테이블 생성 시 캐싱 옵션을 켜면 쓰기도 로컬·원격 양쪽에 함께 발생.

2.6 쿼리 처리 흐름

  1. 클라이언트가 MySQL 프로토콜로 FE에 접속.
  2. FE 파서가 SQL → AST → 논리 플랜 생성.
  3. CBO가 통계를 보고 물리 플랜(Fragment)으로 변환.
  4. FE 코디네이터가 Fragment를 데이터 보유 BE/CN에 분배.
  5. BE/CN이 벡터화 엔진으로 단편을 실행. 필요시 Exchange로 데이터 셔플.
  6. 최종 단편이 결과를 모아 FE를 거쳐 클라이언트에 반환.

클러스터 적용 노트 — 약 20대 PM에서의 노드 배치

FE는 메모리·디스크 사용량은 적지만 안정성이 매우 중요하므로, FE 3대를 별도 PM 3대에 분리하는 게 정석입니다. BE와 같은 PM에 두면 BE가 메모리/CPU를 잡아먹을 때 FE의 응답성이 떨어질 수 있습니다.

  • FE: 3대 (Follower 1 + Follower 2 + Follower 3 / 그중 1대가 Leader). 5대까지 늘릴 필요는 약 20대 규모에선 거의 없음.
  • BE: 나머지 약 17대 모두. storage_root_path에 SSD 8개 경로를 모두 등록.
  • Observer는 BI/대시보드 트래픽이 큰 경우 1~2대 추가 검토(읽기 분산).

장애 시나리오: BE 1대가 죽어도 Tablet 3복제이므로 쿼리는 계속 처리됩니다. FE 1대가 죽어도 Leader는 자동 재선출됩니다(2대 Follower 잔존 → quorum 유지).

2.8 설계 철학

  • 단순함 — 외부 의존 없음. 노드 종류 2~3가지.
  • 현대적 하드웨어를 전제 — SIMD, 큰 메모리, NVMe SSD를 가정한 벡터화 엔진.
  • "하나의 엔진으로 다양한 워크로드" — 칼럼 OLAP인데 UPSERT 가능. 자체 스토리지인데 외부 카탈로그도 흡수.
  • MySQL 친화 — 운영자/분석가의 기존 도구를 그대로 쓰게 함.
