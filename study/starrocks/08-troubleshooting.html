5-1. 트러블슈팅 — 자주 만나는 증상과 해법

BE가 자꾸 죽는다

  • OOM 가능성: be.WARNING / dmesg 확인. mem_limit를 80% → 70%로 내리고 큰 쿼리는 query_mem_limit로 분리.
  • 디스크 가득: SHOW BACKENDS의 MaxDiskUsedPct. 90%를 넘으면 BE가 self-protect로 종료될 수 있음. 파티션 정리/추가 디스크.
  • 네트워크: BE↔BE 9060 포트 유실 → heartbeat fail. 방화벽 변경 직후라면 의심.

쿼리가 갑자기 느려진다

  • 통계 누락/오래됨: ANALYZE TABLE ...;
  • 컴팩션 적체: SHOW PROC '/statistic'; compaction score > 100이면 적체.
  • MV가 무효화됨: SHOW MATERIALIZED VIEW ...; 활성 상태와 마지막 리프레시 시간 확인.
  • Tablet 복구 중: BE 한 대 재기동 직후라면 정상. SHOW PROC '/statistic';의 unhealthy/clone 큐 확인.

메타스토어 변경이 안 보인다 (외부 카탈로그)

REFRESH EXTERNAL TABLE hive_prod.dwh.events;
REFRESH CATALOG hive_prod;

또는 캐시 TTL을 줄이고 백그라운드 자동 갱신을 켭니다(Hive Cache).

HMS 연결 실패

  • 모든 FE/BE에서 thrift://hms:9083 도달 가능한가?
  • Kerberos인 경우 keytab 만료/principal 일치 여부.
  • HMS HA URL 다수 등록 시 한 노드라도 비정상이면 첫 시도 지연 — 정상 노드를 앞쪽에 배치.

"Too many connections"

-- 글로벌 동시 연결 한도
ADMIN SHOW FRONTEND CONFIG LIKE 'qe_max_connection';
ADMIN SET FRONTEND CONFIG ("qe_max_connection" = "4096");

BI 도구의 connection pool 누수도 흔한 원인. SHOW PROCESSLIST에서 idle이 비정상으로 많은지 확인.

잘못 돌고 있는 쿼리 죽이기

SHOW PROCESSLIST;
KILL QUERY <id>;
KILL CONNECTION <id>;

FE 메타데이터 손상

최악의 시나리오. 절차:

  1. 모든 FE 정지.
  2. 이전 백업한 fe/meta를 같은 위치에 복원.
  3. Leader가 될 FE 하나를 먼저 기동, 다른 FE는 --helper로 합류.
  4. SHOW FRONTENDS alive 확인.

적재(Stream/Routine Load) 실패

  • "label already exists": label 충돌. 시간/시퀀스를 label에 포함.
  • "too many filtered rows": 데이터 포맷 문제. SHOW LOAD WHERE LABEL='...'의 ErrorURL에서 실패 행 확인.
  • Routine Load의 PAUSED: SHOW ROUTINE LOAD; 후 RESUME ROUTINE LOAD ...;
클러스터 적용 노트
  • 로그 일원화 필수: 약 20대 PM의 fe/log·be/log를 ELK/Loki에 모으면 트러블슈팅 시간이 절반으로.
  • 증상→체크리스트를 위키로 두고, 알람도 같은 분류로 보내면 신규 운영자가 빨리 적응.
