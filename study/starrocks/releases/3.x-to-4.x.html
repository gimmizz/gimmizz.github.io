3.x → 4.x 업그레이드 체크리스트

⚠️ 메이저 점프 주의

3.x 라인이 너무 오래된(예: 3.1) 경우 한 번에 4.x로 가기보다 3.5 또는 3.6 LTS를 거쳐 단계적으로 가는 게 안전. 메타 호환성·deprecated 함수 정리에 시간이 필요.

사전 준비 (D-7 ~ D-1)

  1. 릴리스 노트 정독 — Behavior change 섹션 집중.
  2. JDK 17 준비 — 모든 FE/BE 노드에 설치, JAVA_HOME 점검.
  3. FE 메타데이터 백업 — tar czf fe-meta-$(date).tgz fe/meta.
  4. 핵심 마트 BACKUP SNAPSHOT — 최소 직전 1주.
  5. 회귀 테스트 SQL 모음 — 자주 도는 BI/ETL SQL 30~50개를 골라 결과 hash 저장.
  6. 새 버전 바이너리 배포 — 모든 PM에 사전 unpack (Ansible).
  7. Deprecated 함수 점검 — audit log 또는 리뷰로.
  8. 변경 PROPERTIES 검토 — replication_num, enable_persistent_index 등.
  9. 알람 일시 무음 — 업그레이드 윈도 동안.
  10. 롤백 계획 — 이전 버전 바이너리 위치 + 스위칭 절차.

업그레이드 순서 (총 1~3시간)

FE

  1. Observer FE → Follower FE → Leader FE 순으로 1대씩.
  2. 각 FE: stop_fe.sh → 새 버전 디렉터리로 심볼릭 링크 변경 → JDK 17 환경변수 → start_fe.sh --daemon.
  3. 다음 FE 진행 전 SHOW FRONTENDS에서 alive·version 확인.

BE

  1. 한 번에 1~2대씩만 (Tablet 3복제 안전).
  2. 각 BE: stop_be.sh → 새 버전 → start_be.sh --daemon.
  3. 다음 BE 진행 전 SHOW BACKENDS에서 alive + clone queue 비워짐 확인: 
    SELECT count(*) FROM information_schema.tablet_schedules
WHERE state IN ('PENDING','RUNNING');

사후 검증

  1. 버전 확인: SELECT version(); 모든 FE/BE.
  2. 회귀 SQL: 사전에 모은 30~50개 SQL 결과 hash 비교.
  3. Routine Load: SHOW ROUTINE LOAD RUNNING 확인.
  4. MV: SHOW MATERIALIZED VIEWS의 is_active / last_refresh_state.
  5. Grafana: p99 query latency, BE memory, compaction score 추적.
  6. 외부 카탈로그: 핵심 카탈로그에서 SELECT 1개씩 — 메타 호환 확인.

롤백

  1. 치명적 회귀 발견 시 — BE 한 대씩 이전 버전 심볼릭 링크로 교체 → 재기동.
  2. FE도 동일. 단 메이저 다운그레이드(예: 4.0 → 3.5)는 메타 호환 문제 가능 — 가능하면 피한다.
  3. 최악의 경우: 업그레이드 직전 fe-meta 백업 복원 + 이전 버전 재기동.
