그림 5-3a-1. Shared-Nothing — 데이터와 컴퓨트가 같은 노드

핵심 원리

  1. 데이터 로컬리티 — Tablet은 BE의 로컬 SSD에 저장. 그 BE가 자기 Tablet을 직접 스캔.
  2. 다중 복제 — 한 Tablet의 N개(보통 3) 복제본은 서로 다른 BE에. 단일 BE 다운에서도 쿼리 가능.
  3. MPP 분산 — FE가 Fragment를 데이터 보유 BE에 분배. BE는 자기 부분만 빠르게.
  4. Exchange로 결합 — 조인/집계 단계에서 BE 간 데이터 셔플(broadcast/hash/bucket).

그림 5-3a-2. Tablet 복제 3개가 서로 다른 BE의 디스크에 분산

쿼리 처리 흐름

sequenceDiagram participant C as Client participant FE as FE participant B1 as BE1 (data A,B) participant B2 as BE2 (data B,C) participant B3 as BE3 (data C,A) C->>FE: SELECT ... GROUP BY ... FE->>FE: Plan: scan + partial agg + final agg FE->>B1: Fragment scan(A,B) + partial_agg FE->>B2: Fragment scan(B,C) + partial_agg FE->>B3: Fragment scan(C,A) + partial_agg B1-->>B2: Exchange (hash by key) B3-->>B2: Exchange (hash by key) B2->>FE: final_agg result FE->>C: rows

강점

  • 저지연·고동시성 — 로컬 SSD에서 직접 스캔. 네트워크 IO 최소.
  • 단순한 구조 — 외부 의존(객체 스토리지, ZK 등) 없음.
  • 강한 데이터 일관성 — Tablet 복제본이 quorum 기반으로 동기화.
  • 풍부한 인덱스 효과 — Zone Map, Bitmap, Bloom, N-Gram이 BE 로컬에서 즉시 동작.
  • 저장과 컴퓨트가 함께 확장 — BE 1대 추가하면 둘 다 ↑.

약점 / 트레이드오프

  • 스토리지·컴퓨트가 묶여 있음 — 데이터가 더 필요해도 컴퓨트가 같이 늘어남(반대도 동일).
  • BE 추가 시 리밸런스 — Tablet이 새 BE로 이동 → 일시적 부하.
  • SSD 비용 — 모든 데이터가 로컬 SSD에 있어야 → GB당 비용이 객체 스토리지보다 비쌈.
  • 장기 보관 비효율 — 콜드 데이터까지 SSD에 두는 건 낭비.

고가용성

장애영향복구
BE 1대 다운해당 Tablet은 다른 2복제본에서 처리. 쿼리는 계속.5분 이상 다운이면 자동 클론 시작 → 새 BE/디스크로 복제본 재생성.
디스크 1개 사망그 디스크 위 Tablet만 영향. BE는 살아있음.해당 Tablet은 다른 BE의 복제본에서 클론.
BE 다수 동시 다운한 Tablet의 모든 복제본이 함께 죽으면 데이터 일시 불가.복제본이 살아있는 BE가 살아나면 자동 복구.
FE 1대 다운Leader 자동 재선출.Follower 다수가 살아있어야.

확장 패턴

수직 확장

  • 같은 BE에 디스크 추가. storage_root_path에 새 경로 추가 후 BE 재기동.
  • 가장 빠르게 디스크 부족을 해소.
  • 한 BE에 디스크가 너무 많아지면(예: >10TB) 복구가 오래 걸려 위험 — 가이드 참고(4-1).

수평 확장

  • BE 추가 (ALTER SYSTEM ADD BACKEND ...).
  • FE 스케줄러가 자동으로 일부 Tablet을 새 BE로 이전(리밸런스).
  • 야간 작업 권장 — 리밸런스가 IO를 점유.

운영 포인트

  • BE 디스크 균형: 디스크별 사용률 편차가 크면 일부가 먼저 가득 참. 정기 점검 필요.
  • Tablet 수: 한 BE당 Tablet이 너무 많으면 메타 부담. 노드당 100K 이하가 일반적.
  • Compaction: 적재가 빈번하면 컴팩션 적체. be_compaction_score 모니터링.
  • 리밸런스 정지: 대량 BE 추가 시 disable_balance=true로 잠시 정지.

언제 Shared-Nothing이 답인가

  • 온프레 자체 데이터센터 + 안정적 SSD 다수 → 이 가이드의 가정 환경.
  • 저지연 BI/대시보드, 고동시성.
  • Hive/Iceberg 외부 카탈로그를 함께 쓰면서 자체 마트는 작은 규모.
  • 운영 인력이 객체 스토리지를 별도로 키우기 어려운 조직.

언제 Shared-Data를 검토

  • 보관 기간이 길고 콜드 데이터 비중이 큼.
  • 야간/낮 부하 차이가 커서 컴퓨트만 확장하고 싶음.
  • 여러 분석 클러스터가 같은 데이터를 공유.
  • 이미 환경에 안정적 S3 호환 스토리지가 정착.
클러스터 적용 노트 — 약 20대 PM 환경
  • SSD 480GB×8 / 노드 = ~3.84TB. 약 17 BE면 raw 65TB → 3복제 환산 약 22TB 가용. Shared-Nothing이 자연스러운 출발점.
  • FE 3 + BE 17 + Observer 1~2(선택)로 시작. 나중에 BI 트래픽 폭증 시 Observer 추가.
  • BE 1대 추가 시 리밸런스가 30분~2시간 소요(데이터 양 따라). 야간 윈도에서 진행.
  • 외부 카탈로그(Hive/Iceberg)를 적극 활용하면 내부 SR 마트가 22TB 안에 충분히 들어감 — 모든 데이터를 내부로 가져가지 말 것.
  • 장기 보관 데이터는 Hive/Iceberg에 두고 SR로는 핫 분석만. 향후 Shared-Data로의 이행은 데이터 부피가 50TB+ 넘어갈 때 검토.
