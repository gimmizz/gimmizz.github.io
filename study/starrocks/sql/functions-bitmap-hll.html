SQL-6. Bitmap · HLL — 정확 / 근사 UV

📌 한 줄 요약

Bitmap은 정수 ID 집합을 압축 저장하는 정확한 distinct 자료구조, HLL은 1% 내외 오차의 근사 distinct. Aggregate 테이블 컬럼 타입으로 직접 저장 가능 → "사전 집계된 UV" 패턴의 핵심.

Bitmap을 Aggregate 테이블에 저장

CREATE TABLE site_uv_d (
  log_date  DATE NOT NULL,
  site_id   INT  NOT NULL,
  uv        BITMAP BITMAP_UNION
)
AGGREGATE KEY(log_date, site_id)
DISTRIBUTED BY HASH(site_id);

-- 적재 (user_id가 BIGINT 같은 정수면 직접)
INSERT INTO site_uv_d
SELECT log_date, site_id, to_bitmap(user_id)
FROM events;

-- 사용
SELECT log_date, bitmap_count(uv) AS uv_cnt
FROM site_uv_d WHERE log_date = '2026-05-07'
GROUP BY log_date;

Bitmap 함수

함수의미
to_bitmap(int_col)정수 → Bitmap (단일 원소)
bitmap_union(b)여러 bitmap 합집합 (집계 함수)
bitmap_count(b)크기
bitmap_and(a,b) / bitmap_or / bitmap_xor교/합/대칭차
bitmap_andnot(a,b)차집합 a − b
bitmap_contains(b, x)포함 여부
bitmap_to_string(b)디버깅용 문자열
bitmap_from_string(s)역방향
bitmap_intersect(b)여러 bitmap 교집합 (집계)

정수가 아닌 ID에 Bitmap을 쓰려면

Bitmap은 정수 기반. UUID·문자열 ID라면 integer mapping 테이블을 만들고 적재 시 매핑하거나, bitmap_hash / bitmap_hash64로 해시값을 사용 (충돌은 통계적으로 무시 가능 수준).

HLL — 근사 UV (가볍고 빠름)

CREATE TABLE site_uv_hll (
  log_date  DATE NOT NULL,
  site_id   INT  NOT NULL,
  uv        HLL HLL_UNION
)
AGGREGATE KEY(log_date, site_id)
DISTRIBUTED BY HASH(site_id);

INSERT INTO site_uv_hll
SELECT log_date, site_id, hll_hash(user_id_str)
FROM events;

SELECT log_date, hll_cardinality(uv)
FROM site_uv_hll WHERE log_date='2026-05-07' GROUP BY log_date;

HLL 함수

함수의미
hll_hash(any_col)임의 타입 → HLL (단일)
hll_union(h)합집합 (집계)
hll_cardinality(h)근사 cardinality
hll_union_agg(h)cardinality(union(h)) 합쳐서 한 번에
approx_count_distinct(col)HLL 기반 즉석 근사

선택 가이드

상황권장
정수 user_id, 정확 UV 필요, 큰 cardinalityBitmap
UUID/문자열 ID, 1% 오차 OKHLL
Cohort 교집합 분석 (어제 + 오늘 사용자 교집합)Bitmap (AND 연산)
그냥 빠른 distinct 추정approx_count_distinct
클러스터 적용 노트
  • 광고/사용자 분석에서 일자별 UV + 누적 UV + 코호트 유지율은 Bitmap이 가장 강력. user_id를 정수로 통일해두는 사전 정비가 핵심.
  • Aggregate 테이블 + BITMAP_UNION 컬럼을 베이스로, 그 위에 Async MV로 한 번 더 압축하는 패턴이 많이 쓰임.
  • HLL은 256GB BE 메모리 환경에선 거의 부담 없음 — 가벼운 대시보드 UV에 기본 선택지.
