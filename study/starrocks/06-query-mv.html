7. 쿼리 · Materialized View

📌 한 줄 요약

StarRocks 쿼리는 벡터화 + CBO + MPP 셔플로 빠르다. 여기에 Async MV를 만들어 두면 사용자는 원본 SQL을 그대로 쓰지만 옵티마이저가 자동으로 MV를 사용한다.

7.1 쿼리 실행 모델

StarRocks의 BE는 다음 단위로 동작합니다.

  • Fragment — 물리 플랜의 한 조각. FE 코디네이터가 BE에 분배.
  • Pipeline — Fragment 안의 실행 단위. BE 내 여러 코어를 활용해 병렬 실행.
  • Operator — Scan, Aggregate, HashJoin, Exchange 등 벡터화 연산자.
  • Exchange — Fragment 간 셔플. Broadcast/Hash Partition/Bucket Shuffle.

7.2 CBO 사용을 위한 통계 관리

CBO는 통계가 있어야 정확한 플랜을 만들 수 있습니다.

-- 수동 통계 수집
ANALYZE TABLE mart.orders;
ANALYZE TABLE mart.orders WITH SAMPLE 100000 ROWS;

-- 통계 조회
SHOW STATS META FOR mart.orders;
SHOW STATS COLUMN FOR mart.orders;

v2.x부터 자동 통계가 활성화되어 있어 일반적으로는 별도 작업이 필요 없지만, 대규모 변경 직후나 성능 회귀 의심 시에는 수동 ANALYZE가 도움이 됩니다.

7.3 JOIN 전략

전략언제 쓰는가비용
Broadcast한쪽 테이블이 매우 작을 때 (예: 차원 테이블)작은 쪽을 모든 BE에 복제
Shuffle (Hash)양쪽 모두 큰 일반 케이스두 테이블 모두 키 기준 셔플
Bucket Shuffle한쪽이 조인 키로 버킷팅된 경우한쪽만 셔플 — 절약
Colocate두 테이블이 Colocate Group으로 같은 분배인 경우셔플 없음 — 가장 빠름

7.4 자주 쓰는 힌트

-- 조인 전략 강제
SELECT /*+ broadcast(b) */ * FROM big a JOIN small b ON a.id = b.id;

-- 셔플 강제
SELECT /*+ shuffle(b) */ * FROM a JOIN b ON a.id = b.id;

-- 특정 MV 사용 강제 (또는 차단)
SELECT /*+ MV_REWRITE_OFF */ * FROM ...;

7.5 EXPLAIN 읽기

EXPLAIN
SELECT dt, count(*) FROM events WHERE dt = '2026-05-07' GROUP BY dt;

-- 더 자세히 (실행 시간 + 메트릭)
EXPLAIN ANALYZE
SELECT dt, count(*) FROM events WHERE dt = '2026-05-07' GROUP BY dt;

읽을 때 주목할 항목:

  • cardinality — 추정 행 수. 통계가 부정확하면 크게 어긋남.
  • Predicates — 필터가 어디까지 푸시다운되었는가.
  • Exchange — 셔플의 종류와 양.
  • RewriteByMV — MV로 자동 재작성되었는지.

7.6 Materialized View — Sync vs Async

Sync MV (단일 테이블 집계 인덱스)

베이스 테이블에 적재가 일어나면 동시에 갱신되는 가벼운 MV. 단일 테이블에 한정됩니다. 

-- 베이스 테이블의 일자별 집계 MV
CREATE MATERIALIZED VIEW mv_event_d AS
SELECT dt, channel, count(*) AS cnt
FROM events
GROUP BY dt, channel;

Async MV (다중 테이블 + 외부 카탈로그까지) ⭐

여러 테이블을 조인하거나 외부 카탈로그를 포함할 수 있고, 스케줄/파티션 기반 리프레시를 지원하는 강력한 MV입니다. Query Rewrite가 핵심으로, 사용자가 원본 SQL을 그대로 써도 옵티마이저가 MV를 알아서 활용합니다. 

CREATE MATERIALIZED VIEW mv_event_user_d
PARTITION BY dt
DISTRIBUTED BY HASH(user_id) BUCKETS 32
REFRESH ASYNC
       START('2026-05-08 02:00:00')
       EVERY (INTERVAL 1 HOUR)
PROPERTIES (
  "replication_num" = "3"
)
AS
SELECT
  e.dt,
  e.user_id,
  u.grade,
  count(*) AS cnt,
  sum(e.amount) AS amt
FROM hive_prod.dwh.events e
JOIN mysql_ops.app.users  u ON e.user_id = u.id
WHERE e.dt >= '2026-01-01'
GROUP BY e.dt, e.user_id, u.grade;

그림 7-1. Async MV의 Query Rewrite — 옵티마이저가 자동 치환

Query Rewrite

옵티마이저가 SQL을 보고 MV로 자동 치환할 수 있는지 판단합니다. 활성화/확인.

-- 세션 변수
SET enable_materialized_view_rewrite = true;

-- MV별 강제 활성화
ALTER MATERIALIZED VIEW mv_event_user_d SET ("enable_query_rewrite" = "true");

-- EXPLAIN 결과에서 'RewriteByMV' 항목 확인

리프레시 모드

  • ASYNC + 스케줄(EVERY) — 정기 리프레시.
  • ASYNC + 수동(REFRESH MATERIALIZED VIEW mv_x;) — 외부 트리거.
  • MANUAL — 자동 리프레시 안 함, 명시적으로만.
  • INCREMENTAL(파티션 단위) — 베이스 테이블에서 변경된 파티션만 리프레시. 비용 큰 MV에서 필수.

7.8 자주 쓰는 운영 SQL 모음

-- 활성 쿼리
SHOW PROCESSLIST;
SHOW PROC '/current_queries';

-- 쿼리 강제 종료
KILL QUERY <id>;

-- 최근 쿼리 통계 (Audit log 플러그인 적재 후)
SELECT user, COUNT(*) cnt, AVG(query_time)/1000 avg_sec
FROM starrocks_audit_db__.starrocks_audit_tbl__
WHERE event_time >= NOW() - INTERVAL 1 HOUR
GROUP BY user
ORDER BY cnt DESC;

-- MV 상태
SHOW MATERIALIZED VIEWS;
SHOW MATERIALIZED VIEW mv_event_user_d\G

-- MV 강제 리프레시
REFRESH MATERIALIZED VIEW mv_event_user_d;
REFRESH MATERIALIZED VIEW mv_event_user_d
        PARTITION START('2026-05-07') END('2026-05-08')
        FORCE;
클러스터 적용 노트 — MV 운영 패턴
  • "BI 응답이 느리다 → MV로 캐시"가 첫 번째 대응. 자주 쓰는 결합/집계를 MV로 떠두면 외부 카탈로그 위에서도 BI 대시보드가 즉시 응답합니다.
  • 새벽 리프레시 정책: 무거운 MV는 새벽(02~04시)에 한 번 정리. EVERY (INTERVAL 1 DAY)로 시작하고, 데이터 신선도 요구가 늘면 시간 단위로 줄입니다.
  • Incremental 우선: 파티션 단위 베이스 테이블이라면 처음부터 INCREMENTAL로 만든다. 풀 리프레시는 점점 비싸집니다.
  • MV의 복제본은 BE 수에 맞게 3: 약 17 BE 환경에서 3복제는 안전. MV가 너무 커서 한 BE의 디스크를 위협한다면 파티션을 잘게 쪼개고 정렬키를 쿼리 패턴에 맞게 둡니다.
  • Resource Group으로 MV 리프레시 격리: MV 리프레시 잡을 별도 Resource Group으로 묶으면 사용자 쿼리에 미치는 영향이 작아집니다.
