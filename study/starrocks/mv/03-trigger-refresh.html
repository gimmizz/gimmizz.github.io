7-3. MV 리프레시 모드

모드 비교

모드의미예시
ASYNC + EVERY주기적 자동 리프레시REFRESH ASYNC EVERY (INTERVAL 1 HOUR)
ASYNC (수동)외부 트리거REFRESH MATERIALIZED VIEW mv_x;
MANUAL자동 안 함, 명시적으로만스케줄러가 따로 호출
INCREMENTAL변경된 파티션만(파티션 정합 필수)

ASYNC + EVERY (가장 흔한 패턴)

CREATE MATERIALIZED VIEW mv_event_d
PARTITION BY dt DISTRIBUTED BY HASH(user_id)
REFRESH ASYNC
       START('2026-05-08 02:00:00')
       EVERY (INTERVAL 1 HOUR)
PROPERTIES (
  "partition_refresh_number" = "10",
  "auto_refresh_partitions_limit" = "30"
)
AS SELECT date_trunc('day', event_time) dt, user_id, count(*) cnt
   FROM events GROUP BY 1, 2;

MANUAL

CREATE MATERIALIZED VIEW mv_x ...
REFRESH MANUAL
AS SELECT ...;

-- 외부 스케줄러(예: Airflow, Dagster)가 호출
REFRESH MATERIALIZED VIEW mv_x;
REFRESH MATERIALIZED VIEW mv_x PARTITION START('2026-05-07') END('2026-05-08');

INCREMENTAL

-- 베이스에서 변경된 파티션 단위만 다시 빌드
REFRESH MATERIALIZED VIEW mv_event_d FORCE;          -- 강제 풀
REFRESH MATERIALIZED VIEW mv_event_d                 -- 변경 파티션 자동 감지
        PARTITION START('2026-05-07') END('2026-05-08');

운영 SQL

-- 모든 MV
SHOW MATERIALIZED VIEWS;

-- 상세
SHOW MATERIALIZED VIEW mv_event_d\G

-- 리프레시 작업 이력
SELECT * FROM information_schema.task_runs
WHERE definition LIKE '%mv_event_d%'
ORDER BY create_time DESC LIMIT 20;

-- 활성/비활성
ALTER MATERIALIZED VIEW mv_event_d ACTIVE;
ALTER MATERIALIZED VIEW mv_event_d INACTIVE;

-- 정의 변경 (자주 쓰지 않음 — 새 MV 만드는 게 더 안전)
ALTER MATERIALIZED VIEW mv_event_d
SET ("partition_refresh_number" = "20");

-- 삭제
DROP MATERIALIZED VIEW mv_event_d;

자동 트리거 조건

  • EVERY 주기 도래 — 가장 일반.
  • 베이스 변경 감지 (Internal 테이블) — Async MV는 베이스 적재 후 다음 주기에 갱신.
  • 외부 카탈로그 베이스는 메타 변경을 감지해야 — force_external_table_query_rewrite·메타 캐시 갱신.

Resource Group으로 리프레시 격리

ALTER MATERIALIZED VIEW mv_event_d
SET ("session.resource_group" = "rg_etl");

대용량 MV 리프레시가 BI 트래픽을 잡아먹는 사고를 예방.

클러스터 적용 노트
  • 새벽 2~4시 정기 리프레시 + 시간 단위 incremental 조합이 가장 안정적.
  • 외부 트리거가 필요한 경우 MANUAL + Airflow/Dagster — 적재가 끝난 직후 MV refresh를 트리거.
  • 실패한 task는 information_schema.task_runs의 state='FAILED'를 모니터링.
