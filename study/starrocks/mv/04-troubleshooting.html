7-4. MV 트러블슈팅 — "왜 안 쓰이나" 진단

1) MV 상태부터 확인

SHOW MATERIALIZED VIEWS WHERE NAME = 'mv_x'\G
-- 'state' 가 ACTIVE 인가?
-- 'is_active' 가 true 인가?
-- 'last_refresh_state' 가 SUCCESS 인가?

state가 INACTIVE이면 베이스 변경/스키마 차이 등으로 무효화된 상태. ALTER MATERIALIZED VIEW mv_x ACTIVE;로 다시 켜되, 원인부터 파악.

2) EXPLAIN으로 rewrite 여부

SET enable_materialized_view_rewrite = true;
EXPLAIN <사용자 SQL>;
-- "RewriteByMV" 또는 "MaterializedView" 표시가 있어야 함.

3) 자주 보는 원인

A. WHERE 범위 불일치

MV의 WHERE이 사용자 쿼리보다 좁으면 사용 불가. 가급적 MV에는 WHERE를 넣지 말고, 데이터 양은 파티션·TTL로 통제.

B. JOIN 종류/조건 불일치

LEFT JOIN으로 만든 MV에 INNER JOIN 쿼리, 또는 ON 조건이 살짝 다른 경우 → rewrite 안 됨.

C. AVG 직접 저장

MV에 AVG를 직접 저장했다면 롤업 불가. SUM/COUNT로 분해 저장 → 사용자 쿼리에서 sum(sum_amt)/sum(cnt) 형태.

D. 비결정적 함수

RAND(), NOW(), CURRENT_USER() 등은 rewrite 차단. MV 정의에서 제거.

E. stale 파티션

베이스가 변경됐지만 MV가 따라잡지 못한 경우. SHOW PARTITIONS FROM mv_x의 lastRefreshTime을 확인하고 강제 리프레시.

F. 외부 카탈로그 + 메타 캐시

Hive/Iceberg 베이스에서 새 파티션이 추가됐는데 메타 캐시가 안 갱신 → MV가 인식 못 함.

REFRESH CATALOG hive_prod;
REFRESH MATERIALIZED VIEW mv_x;

G. 외부 카탈로그 rewrite 명시 옵션

ALTER MATERIALIZED VIEW mv_x
SET ("force_external_table_query_rewrite" = "true");

4) 리프레시 잡 이력 확인

SELECT task_name, state, error_message, finish_time
FROM information_schema.task_runs
WHERE task_name LIKE '%mv_x%'
ORDER BY create_time DESC LIMIT 20;

5) FE 로그

Rewrite 실패 사유는 fe.log의 INFO/WARN 레벨에 종종 단서가 있음. 키워드: Materialized view, rewrite, MV.

6) MV가 쓰이는데도 느린 경우

  • MV의 분배 키가 쿼리 GROUP BY/JOIN 키와 일치하는가? 안 맞으면 셔플 발생.
  • MV의 정렬키(ORDER BY)가 쿼리 WHERE 조건에 맞는가?
  • MV 통계가 최신인가? ANALYZE TABLE mv_x;

진단 체크리스트

  1. SHOW MATERIALIZED VIEW에서 state·is_active·last_refresh_state 확인
  2. EXPLAIN에서 RewriteByMV 표시 확인
  3. WHERE/JOIN/AVG/비결정 함수 점검
  4. SHOW PARTITIONS로 stale 여부
  5. 외부 카탈로그라면 REFRESH CATALOG + force 옵션
  6. task_runs에서 실패 잡 검토
  7. fe.log에서 rewrite 키워드
  8. 분배 키·정렬키·통계 검토
클러스터 적용 노트
  • "MV가 안 쓰여요" 문의가 운영 단골. 본 페이지를 위키에 그대로 박아 사용자 self-serve 진단을 유도.
  • 큰 MV가 여러 개라면 dashboard에 last_refresh_state stale 알람을 두면 좋습니다.
