MIG-1. MySQL → StarRocks

의도

분석성 워크로드만 SR로 옮기고, 트랜잭션 OLTP는 MySQL에 그대로 두는 게 일반적. SR로 이전하는 것은 "보고서/대시보드/로그 분석"이지 OLTP 본체가 아님.

두 가지 이전 옵션

A. JDBC Catalog로 "보고만"

이전 부담이 가장 작음. MySQL 그대로 두고 SR이 카탈로그처럼 읽음.

CREATE EXTERNAL CATALOG mysql_app
PROPERTIES ("type"="jdbc",
            "user"="sr_reader","password"="...",
            "jdbc_uri"="jdbc:mysql://mysql:3306",
            "driver_url"="file:///opt/sr/drivers/mysql-connector-j-8.4.0.jar",
            "driver_class"="com.mysql.cj.jdbc.Driver");

-- 자주 쓰는 결합은 Async MV로 캐시
CREATE MATERIALIZED VIEW mv_user_d ...
AS SELECT ... FROM mysql_app.app.users JOIN ...;

B. CDC로 SR 내부 마트로 옮기기

실시간 분석/고동시성이 필요할 때.

-- Flink CDC + StarRocks Connector (PK 테이블 자동 UPSERT)
CREATE TABLE src (...) WITH ('connector'='mysql-cdc', ...);
CREATE TABLE snk (...) WITH ('connector'='starrocks', ...);
INSERT INTO snk SELECT * FROM src;

초기 백필

-- A. JDBC Catalog로 한 번 INSERT
INSERT INTO default_catalog.mart.orders
SELECT * FROM mysql_app.app.orders;

-- B. mysqldump → CSV → Stream Load 또는 INSERT FROM FILES()
SELECT count(*) FROM FILES("path"="hdfs://nn/dump/orders/*.csv","format"="csv");

SQL 변환 체크리스트

  • ON DUPLICATE KEY UPDATE → PK 테이블의 자동 UPSERT 또는 UPDATE.
  • 저장 프로시저/트리거 → 애플리케이션 또는 ETL 스케줄러.
  • 큰 OFFSET 페이지네이션 → 키 기반 페이지네이션.
  • FOR UPDATE / 잠금 SELECT → 분석 워크로드에서는 일반 SELECT로.
  • VIEW의 SQL 호환성 점검 — 함수 시그니처 일부 차이.

컷오버 전략

  1. SR 클러스터 + JDBC Catalog 등록 → 분석가 일부 트래픽 이동(Read-only).
  2. 핫한 테이블 SR 내부로 적재 (CDC 또는 일배치) → 응답 비교.
  3. BI 대시보드의 데이터 소스를 SR로 전환.
  4. MySQL 분석 트래픽 모니터링 — 0이 되면 분석용 권한 회수.
  5. OLTP 본체는 MySQL에 그대로 둔다.
클러스터 적용 노트
  • MySQL의 무거운 GROUP BY 대시보드는 SR이 가장 큰 가치를 발휘하는 지점.
  • 이전 단계에서 데이터 일관성 검증을 위해 동일 SQL을 양쪽에서 돌려 결과 diff 검사 — 자동화 권장.
  • 파티션 키와 분배 키를 신중히 선택 — 처음 잘못 잡으면 재구축 비용 큼.
