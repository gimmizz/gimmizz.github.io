MIG-2. ClickHouse → StarRocks

왜 옮기나

  • 분산 JOIN 한계 → SR의 MPP + CBO로 다중 조인 강화.
  • UPSERT/DELETE → SR의 Primary Key 모델.
  • 외부 카탈로그(Lake) 통합 → SR의 다양한 External Catalog.
  • MV 자동 재작성 → CH의 Projection보다 유연.

테이블 엔진 매핑

ClickHouseStarRocks
MergeTreeDuplicate Key 또는 Aggregate Key
ReplacingMergeTreePrimary Key 또는 Unique Key
SummingMergeTreeAggregate Key (SUM)
AggregatingMergeTreeAggregate Key (BITMAP_UNION/HLL_UNION 등)
Distributed EngineSR 자체가 분산 — 별도 엔진 불필요
Materialized ViewSync MV / Async MV
Dictionary차원 테이블 + Colocate Group

스키마 변환 예

-- ClickHouse
CREATE TABLE events
(
  event_time DateTime,
  user_id    UInt64,
  event_type LowCardinality(String),
  url        String
)
ENGINE = MergeTree
PARTITION BY toYYYYMMDD(event_time)
ORDER BY (event_time, user_id);

-- StarRocks 변환
CREATE TABLE events (
  event_time DATETIME NOT NULL,
  user_id    BIGINT,
  event_type VARCHAR(32),    -- 자동 dictionary 처리 (저카디널리티)
  url        STRING
)
DUPLICATE KEY(event_time, user_id)
PARTITION BY date_trunc('day', event_time)
DISTRIBUTED BY HASH(user_id)
PROPERTIES ("replication_num"="3", "compression"="ZSTD");

SQL 다이얼렉트 차이

  • arrayMap·arrayFilter같은 람다 → SR은 일반 SQL/UDF.
  • uniq/uniqExact → approx_count_distinct 또는 BITMAP.
  • quantile → percentile_approx.
  • SAMPLE → SR은 ANSI 표준 TABLESAMPLE 일부 지원.
  • FINAL 키워드(Replacing 강제 머지) → SR은 PK 테이블이라면 불필요.

데이터 이전

# ClickHouse → parquet
clickhouse-client --query="SELECT * FROM events WHERE dt='2026-05-07' FORMAT Parquet" \
  > /tmp/events_20260507.parquet

# parquet → S3/HDFS 적재 후 SR INSERT
INSERT INTO mart.events
SELECT * FROM FILES("path"="s3://bucket/events_20260507.parquet","format"="parquet", ...);

Dual-write로 안전한 컷오버

  1. 적재 파이프라인을 CH와 SR에 동시 적재(Dual write).
  2. 주요 SQL 결과 비교 (자동화).
  3. BI 트래픽을 SR로 점진 이동 — Canary.
  4. 한 달 안정 운영 후 CH 적재 중단, 그 후 read-only 보존 → 폐기.
클러스터 적용 노트
  • CH의 LowCardinality/Compact MergeTree 패턴은 SR에서 별도 노력 없이 자동 처리.
  • 분산 JOIN 워크로드는 SR이 압도적으로 안정적 — 가장 큰 이전 동기.
  • CH의 시스템 함수 의존(예: arrayMap)이 많은 ETL은 dbt-style 표준 SQL로 재작성 권장.
