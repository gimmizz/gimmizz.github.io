MIG-4. Apache Doris → StarRocks

StarRocks와 Apache Doris는 같은 뿌리(Apache Doris fork에서 분기)에서 출발해 SQL/스토리지 포맷이 매우 유사합니다. 그래서 마이그레이션이 가장 쉬운 케이스.

매우 유사한 부분

  • FE/BE 아키텍처, MySQL 프로토콜.
  • Duplicate/Aggregate/Unique/Primary Key 모델.
  • Stream Load / Routine Load / Broker Load.
  • BACKUP/RESTORE 개념.
  • 대부분의 SQL 함수.

스키마 변환

대부분의 CREATE TABLE은 그대로 또는 거의 그대로 동작. 다음만 주의:

  • 일부 PROPERTIES 키 이름 차이 (예: dynamic_partition 옵션).
  • Primary Key 테이블의 enable_persistent_index는 SR 전용.
  • 분배: SR은 Auto Bucket이 더 적극적.

데이터 이전 옵션

  1. BACKUP/RESTORE: Doris의 SNAPSHOT을 SR이 직접 복원하는 건 호환성 보장 안 됨 — 권장하지 않음.
  2. SELECT INTO OUTFILE → 적재: parquet/csv로 export 후 SR에서 INSERT FROM FILES().
  3. JDBC Catalog로 Doris 직접 SELECT → SR INSERT: 단발 백필에 가장 편함.
  4. Dual write: 적재 파이프라인을 양쪽에 동시 흘려 안전한 컷오버.
-- 옵션 3 예시
CREATE EXTERNAL CATALOG doris_old
PROPERTIES ("type"="jdbc",
            "user"="reader","password"="...",
            "jdbc_uri"="jdbc:mysql://doris-fe:9030",
            "driver_url"="file:///opt/sr/drivers/mysql-connector-j-8.4.0.jar",
            "driver_class"="com.mysql.cj.jdbc.Driver");

INSERT INTO default_catalog.mart.orders
SELECT * FROM doris_old.mart.orders;

BI 도구 마이그레이션

Doris도 MySQL 프로토콜이라 BI 도구 설정의 호스트/포트만 바꾸면 그대로 동작. 권한·역할은 RBAC 기반으로 재구성.

체크리스트

  1. SR 클러스터 구축 + 같은 외부 카탈로그 등록.
  2. Doris의 DDL을 mysql 클라이언트로 dump → 약간 수정 후 SR에서 실행.
  3. JDBC Catalog로 Doris 데이터 INSERT 백필.
  4. 적재 파이프라인을 SR에도 흘리고 결과 비교.
  5. BI 데이터 소스 전환.
  6. Doris 적재 중단 → 보관 → 폐기.
클러스터 적용 노트
  • Doris ↔ SR은 마이그레이션 부담이 가장 작은 케이스. 일반적으로 1~2주에 핵심 마트 이전 가능.
  • 이전 후 SR의 추가 가치(다양한 외부 카탈로그, MV Rewrite, Data Cache)를 적극 활용하면 ROI가 극대화.
