MIG-3. Trino / Presto → StarRocks

왜 옮기나

  • 스토리지 + 컴퓨트가 한 엔진 → BI 가속용 자체 마트 가능.
  • Async MV 자동 재작성 → 사용자 코드 변경 없이 가속.
  • Data Cache → 외부 카탈로그가 내부에 가까운 응답 속도.
  • UPSERT/DELETE → PK 모델로 가능 (Trino는 커넥터 의존).

SQL 다이얼렉트 차이

  • SELECT * FROM catalog.schema.table 동일 (3단 식별자).
  • WITH (table_property = ...) 같은 Trino DDL 옵션 → SR의 PROPERTIES.
  • UNNEST(array_col) → SR도 지원.
  • GROUP BY GROUPING SETS → 동일.
  • WINDOW 절 명명 → SR도 지원.
  • VARCHAR 파라미터 길이 → SR은 길이 명시 필요.
  • BIGINT/INTEGER 차이 — SR은 BIGINT 표준.

전환 전략

  1. SR 클러스터 구축 + Trino와 같은 메타스토어/스토리지에 카탈로그 등록.
  2. 같은 SQL을 양쪽에서 돌려 결과/응답 시간 비교 (자동 회귀 테스트).
  3. BI 도구의 데이터 소스를 SR로 전환 (선택 SQL부터 단계적).
  4. 핫 SQL은 SR에 Async MV로 캐시 → 더 빠른 응답.
  5. Trino 트래픽이 줄어들면 클러스터 축소.

주의사항

  • Trino-only 함수(특정 Geo, ML)가 있으면 SR UDF/UDAF로 포팅 필요.
  • Trino의 Federation은 매우 강력 — SR도 비슷하지만 확장 가능 카탈로그가 다를 수 있어 검증.
  • Trino의 fault-tolerant execution(예: 큰 ETL)을 의존했다면 SR에선 spill + 재시도 전략.
클러스터 적용 노트
  • Trino를 BI 가속에 쓰던 워크로드가 가장 큰 이전 효과. 카탈로그 등록 + Data Cache + Async MV가 표준 셋업.
  • 데이터 사이언스가 Trino의 Federation을 SQL로 많이 쓰던 경우, SR Unified Catalog로 동등 이상의 경험 제공 가능.
  • SR로 옮기고도 여전히 ad-hoc large scan이 필요하다면 Trino를 배치 분석에 남겨두고 SR은 BI/대시보드 전용으로 분리하는 하이브리드도 OK.
