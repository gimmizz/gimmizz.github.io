PAT-3. Lakehouse 가속 (Trino/Presto 대체)

목표

이미 Hive/Iceberg/Hudi/Delta로 만들어진 데이터 레이크 위에서, 적재 없이 빠른 분석을 제공. Trino/Presto 대안.

Data Cache 활성

# be.conf
datacache_enable    = true
datacache_disk_path = "/data/sr/cache"
datacache_disk_size = "400G"
datacache_mem_size  = "10%"

자주 쓰는 분석을 MV로

CREATE MATERIALIZED VIEW mv_lake_event_d
PARTITION BY dt
DISTRIBUTED BY HASH(user_id)
REFRESH ASYNC EVERY (INTERVAL 1 HOUR)
PROPERTIES (
  "enable_query_rewrite"="true",
  "force_external_table_query_rewrite"="true"
)
AS
SELECT date_trunc('day', event_time) dt, user_id, event_type, count(*) cnt
FROM hive_prod.dwh.events
GROUP BY 1, 2, 3;

Lake ↔ 내부 마트 결합

SELECT u.grade, count(distinct e.user_id) AS uniq
FROM hive_prod.dwh.events e
JOIN default_catalog.mart.dim_user u ON e.user_id = u.user_id
WHERE e.dt = '2026-05-07'
GROUP BY u.grade;

단계 전략

  1. Hive Catalog 등록 + 분석가에게 SQL 권한 — 즉시 사용 가능.
  2. BI 응답이 느린 SQL Top-N 추출 → MV 후보화.
  3. MV 만들고 enable_query_rewrite=true로 사용자 코드 변경 없이 가속.
  4. Data Cache 도입 후 적중률 모니터링 (be_data_cache_hit_ratio).
  5. 특별히 핫한 테이블만 내부로 INSERT — 나머지는 외부에 둠.

클러스터 적용 노트
  • 약 17 BE 환경에서 SSD 1개를 Data Cache 전용(약 480GB)으로 떼면 17 × 480 = ~8TB의 핫 캐시 — 직전 1~2주 파티션은 항상 로컬에서 응답.
  • Lakehouse의 데이터 정합 책임은 원천에 — Hive/Iceberg 측 컴팩션과 메타 정책 점검이 안정성 핵심.
  • Trino를 운영 중이라면 한쪽에 SR을 함께 두고 단계적으로 트래픽 이동하는 패턴이 안전.
